# v1.2 — Polish, Intelligence & User Experience — Report

## 1. Summary

v1.0 (pipeline stabilization) and v1.1 (ecosystem expansion) were already stable and are unchanged by this pass. This session focused on high-impact, low-complexity improvements rather than new subsystems: one real bug fix, one new deterministic job-intelligence extractor, a genuinely missing settings UI for an already-shipped backend feature, Telegram message polish, and dev-experience scripts. Every change is additive (new nullable columns, new optional preference fields, new npm scripts) — nothing existing was redesigned.

**Quality gates, run at the end of every phase:** `npx tsc --noEmit` clean · `npx vitest run` → 649/649 passing (75 files) · `npm run build` succeeds. `npm run verify` now runs all three as one command.

## 2. Implementation by phase

### Phase 1 — Bug Hunt
Found and fixed one real bug: `filterMatches.ts`'s skill filter checked only `description`, while `scoreJob.ts` (the source of truth for scoring) checks `title + description`. A job that scored well on a title-only skill mention could be silently dropped by a notification skill filter. Fixed + regression-tested. Full findings in `docs/reviews/2026-07-04/technical-debt.md`.

### Phase 2 — Job Intelligence
Added `src/features/jobs/domain/extractJobAttributes.ts` — a deterministic, regex-only extractor (same style as the existing `extractSalary.ts`/`extractContactEmail.ts`) producing, from title+description at ingest:
- `employmentType`: internship | contract | freelance | temporary | part_time | full_time
- `seniority`: executive | principal | lead | senior | entry | mid
- `workArrangement`: hybrid | onsite (remote is already covered by `location_tags`)
- `visaSponsorship` / `relocationAssistance`: tri-state (null = not mentioned, true/false = explicit)
- `securityClearance` / `urgentHiring`: boolean

Wired into `ingestJobs.ts` alongside the existing salary/contact-email extraction. New migration `20260704000003_job_attributes.sql` adds the 7 columns plus an index on `employment_type`. 30 new unit tests, all passing on first run.

### Phase 3 — Personal Intelligence
Extended `NotificationPreferences` with `blockedCompanies` (substring match against company name) and `excludeEmploymentTypes` (using Phase 2's new field) — both exclude filters, ANDed after the existing include filters. Added `validateNotificationPreferences` (rejects unknown enum values and an inverted min/max experience range).

**The bigger find:** `NotificationPreferences` (roles/skills/locations/sources/experience filters) shipped in an earlier phase with full domain/application/infrastructure support and a server action — but **no UI**. The only way to set it was a direct `app_settings` write. Built `NotificationPreferencesCard` on `/settings` so the feature is actually usable end-to-end for a single daily user, which is the core ask of this mission ("remove friction... prefer simple configuration").

### Phase 4 — Smart Notifications
Added `buildJobHighlights.ts` — a pure function producing short "why this job" badges (🌍 Remote, ⚡ Urgent hiring, 💰 salary range, 📄 employment type for non-full-time roles) from data already computed at ingest, no extra AI calls. Wired into both live message formats: `formatMatchMessage.ts` (individual mode, default) and `formatDigestMvp.ts` (digest mode, both the strong-match list and the worth-reviewing follow-up). The legacy `formatDigestMessage.ts` (`digest_legacy` mode, not the default) was deliberately left untouched — no reason to invest further in a path already superseded by the MVP digest.

### Phase 7 — Developer Experience
- `npm run verify` — composite `typecheck && test && build`, the single command that maps directly onto this mission's own Quality Gates.
- `npm run doctor` (new `scripts/doctor.ts`) — checks all required/optional env vars are set, then does one live Supabase query and one Telegram `getMe` call. Verified it correctly reports 8 failing / 7 warning when run with no env configured, and exits non-zero.
- `npm run health` — alias for `validate-sources` under the name this mission's own vocabulary uses.
- `npm run diagnose` — chains the two previously-unwired diagnostic scripts (`report-sources.ts` + `filter-analysis.ts`).
- `npm run analytics` / `npm run report:sources` / `npm run backfill:min-years` — three more scripts that existed on disk but had no `package.json` entry, now discoverable.

### Phases 5, 6, 8 — not undertaken this pass
Dashboard UI (Phase 5) and source-quality ranking (Phase 6) were reviewed but not changed: the dashboard already has solid filtering/sorting, and source health/quality is already extensively covered (`getSourceHealthReport`, `/analytics`, `validate-sources`, the newly-wired `analytics`/`diagnose` scripts) per `design/limitations.md` §8's own existing backlog. Repository cleanup (Phase 8) was limited to what Phase 1's audit actually turned up (see technical-debt.md) — no speculative renames or reorganization, per the mission's own "do not refactor working code without measurable benefit" instruction.

## 4. Rationale

Every change maps to an existing pattern already in the codebase (extractor style, JSON-blob preference storage, settings-card UI style, npm-script wiring) — no new architecture, no new dependencies, no Prisma/Drizzle/Zustand/Redux/React Query. The domain→application→infrastructure→tests ordering was followed for both new features before any UI was touched (job attributes: domain extractor + migration + repo wiring + tests, then later consumed by notifications; notification preferences: domain type + validation + filter logic + tests, then the settings card).

## 6. Risks

- **Migration not yet applied.** Like every prior phase, `20260704000003_job_attributes.sql` needs `supabase db push` (and `supabase gen types` to replace the manually-synced `database.types.ts` edits made this pass) before the new columns exist in a live database. Until applied, `ingestJobs`'s new fields will be silently ignored/error depending on Postgres's tolerance for extra insert columns — this is the same "DB not migrated yet" risk every prior migration-adding phase has carried (see `docs/plans/STATUS.md`).
- **Regex-based extraction is best-effort by design** (documented extensively in `design/limitations.md` §1.11) — false negatives are expected and preferred over false positives, consistent with `extractSalary`/`extractContactEmail`'s existing precedent.
- **No live verification of the Telegram/dashboard UX** was possible in this sandbox (no Supabase/Telegram credentials configured) — `npm run doctor` confirms this is the actual state of the environment rather than a gap in testing effort. The settings card was verified via `npm run build` (compiles, renders as a server-then-client component correctly) and the underlying preference/filter logic via unit tests, but not click-tested in a running browser against a real database.

2. Preference-based "why this job" richness: preferred companies/technologies/salary threshold, surfaced as a genuine explanation line rather than just badges.

4. Revisit AD-23's deferred batching/adaptive AI model routing now that Phase 2's extra ingest-time regex work adds negligible cost (all deterministic, no new AI calls anywhere this pass).

## 9. Honest assessment

**Excellent:** the domain/application/infrastructure layering is genuinely consistent — every new extractor, filter, and repository method this session slotted into an existing, obvious location without any architectural judgment calls needed. The design docs are unusually thorough and were easy to extend because prior phases kept them honest.

**Good:** test coverage is strong and the team (prior sessions) clearly already ran a real hardening pass — this session's own bug hunt found exactly one issue in a codebase this size, which is a good sign, not a sign the hunt was shallow.

**Still needs work:** the gap between "backend feature shipped" and "user can actually configure it" (the notification-preferences UI gap this session closed) suggests it's worth periodically auditing for other shipped-but-not-exposed features. `docs/database.md` vs `design/erd.md` drift is a recurring low-grade risk if only one gets updated per phase.

**Should never be changed:** the Repository Pattern discipline, the deterministic-over-AI bias for extraction, and the "fail visible, not silently wrong" convention (AD-16/AD-20/AD-22) — these are why this codebase is pleasant to extend.

**Technical debt:** the items in `technical-debt.md` (stale docs/database.md, two scripts bypassing the repository pattern, per-file test fixture duplication) are all low-priority and explicitly not urgent.

**Feature-complete for personal use?** Yes, with the caveat that the migration in this session (like several before it) still needs to be applied to a live database before it takes effect. The pipeline (scrape → score → notify) is robust, observable, and now has a genuine environment doctor to catch misconfiguration before a cron run fails silently. The remaining backlog is entirely "nice to have," not "broken."
