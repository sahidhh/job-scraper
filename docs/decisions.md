# Architectural Decisions

Numbered for reference (ADR-style). Each records the decision, why, alternatives considered, and consequences.

---

### AD-01: Single-user app with simple Supabase Auth (email/password)

**Decision:** One user account, Supabase Auth email/password, no `user_id` columns anywhere.

**Rationale:** This is a personal tool. Multi-tenancy adds `user_id` to every table, RLS policies per table, and signup/invite flows for zero benefit. A bare no-auth option was rejected because the dashboard will be deployed publicly on Vercel — a login screen is the minimum needed to keep it private.

**Alternatives considered:** No auth at all (rejected — public URL with no protection); full multi-tenant Supabase Auth (rejected — unnecessary complexity for one user).

**Consequences:** Schema is simpler (no ownership columns/joins). If multi-user is ever needed, every table needs a migration to add `user_id` plus RLS rewrites — accepted as unlikely and out of scope.

---

### AD-02: Feature-based folders, clean-architecture layers *within* each feature

**Decision:** `src/features/<feature>/{domain,application,infrastructure}` instead of top-level `src/{domain,application,infrastructure}/<feature>`.

**Rationale:** Top-level layer folders scale poorly — each grows into a flat bag of unrelated feature code, and understanding one feature means jumping between three distant directories. Feature-first folders keep everything about "scoring" or "roles" colocated, while still enforcing the domain/application/infrastructure dependency direction (AD-03) within each feature.

**Alternatives considered:** Classic top-level clean architecture layers (rejected — poor cohesion at this project's size); no layering at all, just feature folders with mixed concerns (rejected — violates "repository pattern" and "no duplicated logic" requirements, makes use-cases untestable without a live Supabase connection).

**Consequences:** Slightly more directories per feature (3 vs 1), but each is small and each feature is independently understandable.

---

### AD-03: Repository pattern — interfaces in `domain`, Supabase implementations in `infrastructure`

**Decision:** Every feature needing persistence defines a `XRepository` interface in `domain/`; `application` use-cases depend on the interface; `infrastructure` provides the Supabase-backed implementation; `presentation` (pages, server actions, `scripts/*.ts`) is the composition root that wires concrete implementations into use-cases.

**Rationale:** Decouples business logic (scoring math, role expansion, location filtering, dedup) from Supabase specifics, enabling unit tests with mock repositories. Also directly satisfies the "repository pattern" and "no duplicated logic" requirements — the dashboard and the cron scripts call the *same* use-cases with the *same* repositories.

**Alternatives considered:** Direct Supabase client calls inside use-cases (rejected — untestable, couples business logic to a specific client/version); a generic ORM-style data layer (rejected — premature abstraction for ~8 tables).

**Consequences:** One interface + one implementation per persisted feature (≈7 repositories total, per `repositories.md`) — a fixed, small amount of boilerplate in exchange for testability.

---

### AD-04: Scraping/scoring/notification run as standalone scripts via GitHub Actions cron, not API routes

**Decision:** `scripts/scrape.ts`, `scripts/score.ts`, `scripts/notify.ts` run via `tsx` in a scheduled GH Actions workflow, using the Supabase service role key directly.

**Rationale:** Decouples the pipeline from the Next.js deployment — it runs whether or not Vercel is up, and isn't subject to serverless function timeout limits (scraping 5 sources × N companies can take longer than typical serverless limits). Scripts import the same `src/features/*` code as the app, so there's no logic duplication despite being a separate entrypoint.

**Alternatives considered:** GH Actions cron hitting Next.js API routes (rejected — adds deployment as a hard dependency for the pipeline and risks serverless timeouts on long scrape runs).

**Consequences:** Two execution contexts to keep in sync on shared config/env handling (`shared/config` reads from `process.env` in both — no special handling needed since both run in Node).

### AD-05: User-maintained `companies` config table for Greenhouse/Lever/Ashby

**Decision:** A `companies` table (editable via `/settings`) lists `(source, board_token)` pairs to scrape. RemoteOK/Wellfound need no such config — they expose feed-style endpoints covering many companies.

**Rationale:** There is no public directory of "every company using Greenhouse/Lever/Ashby" — these ATS's board tokens must be known in advance. A config table is the simplest correct solution and doubles as the mechanism for the user to curate which companies matter to them.

**Alternatives considered:** Attempting to crawl/discover board tokens (rejected — no reliable source, high risk of scraping unrelated/irrelevant company boards).

**Consequences:** The user must add companies manually over time via `/settings`. Coverage of Greenhouse/Lever/Ashby is only as good as this list — an inherent, accepted limitation of these sources for a personal tool.

### AD-06: Role expansion — static map with AI fallback, cached permanently

**Decision:** `role_expansion_map` table seeded with common role clusters; unseen roles trigger one OpenRouter call, and the result is cached (`source='ai'`) so it's never re-requested.

**Rationale:** Satisfies "AI usage minimized" — the AI is a one-time cost *per distinct role string ever entered*, not per role-selection or per cron run. Most usage will hit the seeded map for free.

**Alternatives considered:** Pure static map (rejected — would silently fail/return nothing for roles not anticipated at seed time); AI on every role selection (rejected — wasteful, role choices repeat over time).

**Consequences:** `role_expansion_map` grows slowly over time as new roles are tried; no cache invalidation/expiry — if related-role conventions "should" change, the cached row must be manually edited/deleted (acceptable for a personal tool, revisit only if it becomes annoying).

### AD-07: Two-stage scoring — keyword overlap gates AI refinement

**Decision:** Stage 1 (keyword/skill-dictionary overlap, `features/scoring`) runs for every candidate job, free and instant. Stage 2 (OpenRouter call) runs only when `keyword_score >= KEYWORD_THRESHOLD` (default 0.5).

**Rationale:** Directly implements "OpenRouter only used for high-confidence matches" — stage 1 is the confidence filter. Bounds AI spend to a small, predictable fraction of total scraped jobs (those already skill-relevant), while every job still gets *some* score for dashboard sorting.

**Alternatives considered:** Embedding-based similarity as stage 1 (rejected — adds an embeddings API dependency/cost for marginal accuracy gain over dictionary overlap, given the role-title pre-filter already narrows candidates); AI for every job (rejected — violates "AI usage minimized").

**Consequences:** Jobs with `keyword_score < 0.5` never get an `ai_score`, even if they'd plausibly be a good fit despite low dictionary overlap (e.g. resume/job phrase things differently). Accepted tradeoff — the skills dictionary (AD covers resume parsing too) is the single point of truth for "what counts as a skill," so improving its coverage improves both stages.

### AD-08: Notification idempotency via `notifications_log`, gated on `ai_score`

**Decision:** A job is notified at most once, ever, tracked via a `unique(job_id)` table. Only jobs with `ai_score >= NOTIFY_THRESHOLD` (default 0.75) qualify — `ai_score is null` never qualifies (SQL null comparison is false), regardless of `keyword_score`.

**Rationale:** Prevents duplicate Telegram messages across cron runs (the core idempotency requirement). Gating on `ai_score` specifically (not `keyword_score`) means only AI-refined high-confidence matches page the user — matching the spec's "send Telegram notifications for new matches" in the context of the high-confidence AI scoring stage.

**Alternatives considered:** Gate on `keyword_score` instead (rejected — would notify on the cheap signal alone, defeating the purpose of stage 2); allow re-notification if score improves on a later run (rejected — `job_scores` rows aren't re-computed per role_selection anyway, so "improves" doesn't naturally occur; adds complexity for a case that doesn't arise).

**Consequences:** A job whose stage-2 AI call failed (`ai_score` stays null, per `scoring.md` §3) will never be notified unless a future manual re-score feature is added. Acceptable — AI call failures are expected to be rare (timeout + 1 retry).

### AD-09: "Single active" resume/role_selection via partial unique index + Postgres RPC swap

**Decision:** `resumes.is_active` and `role_selections.is_active` each have a partial unique index (`where is_active = true`). Activating a new row is done via a Postgres function (`set_active_resume`, `set_active_role_selection`) that deactivates the old row then inserts/activates the new one, called via `supabase.rpc()`.

**Rationale:** Keeps history (old resumes/role selections aren't deleted) while guaranteeing exactly one "current" row at the DB level — invariant holds even if the app crashes mid-operation (worst case: zero active rows, recoverable, never two). The Supabase JS client doesn't support multi-statement client-side transactions, so the atomic swap must live in the DB as a function.

**Alternatives considered:** App-level "select then two updates" without a DB function (rejected — a crash between updates could leave two active rows, violating the invariant the unique index is meant to enforce — though the index would actually reject it; more precisely, it could leave the operation half-done with an error and an inconsistent intermediate state); a single `current_resume_id`/`current_role_selection_id` pointer in a settings table instead of `is_active` flags (rejected — slightly more indirection for no real benefit at this scale).

**Consequences:** Two small Postgres functions to maintain as part of the migration set (documented in `database.md` §7, defined in their own migration file).

### AD-10: Wellfound scraper is defensive-by-design, isolated from the rest of the pipeline

**Decision:** The Wellfound adapter validates response shape before mapping and returns `[]` (logging a warning, recording `scrape_runs.status = 'failed'`) rather than throwing, if the unofficial feed's structure doesn't match expectations.

**Rationale:** Wellfound has no documented public API, so its response format is the least stable input to the system. Per-source isolation (AD-04's sibling concern, detailed in `scrapers.md` §4) already prevents one source's failure from blocking others — this decision makes that failure mode *expected and silent* specifically for Wellfound, rather than a recurring error needing investigation.

**Alternatives considered:** Treat Wellfound the same as the ATS sources (rejected — would generate noisy failures disproportionate to the source's value); drop Wellfound entirely (rejected — explicitly requested in the spec).

**Consequences:** If Wellfound's feed structure changes, the system degrades to "zero Wellfound jobs" silently (visible only via `scrape_runs.status='failed'` in `/settings`) rather than failing loudly — accepted given this source is the most likely to need future maintenance regardless.

---

### AD-11: Forward-only migrations, no down-migrations

**Decision:** `supabase/migrations/` contains only forward changes; reverting a change means writing a new forward migration that undoes it.

**Rationale:** Down-migrations are rarely run correctly in practice and add authoring overhead for every migration on a project with one environment (production) and one developer. Forward-only is simpler and matches how the project will actually be operated.

**Alternatives considered:** Standard up/down migration pairs (rejected — overhead not justified for a single-environment personal project).

**Consequences:** No automated rollback path; mistakes are fixed by new migrations. Acceptable given the schema is small and changes are infrequent.

---

### AD-12: RLS enabled with service-role bypass for cron, authenticated-role policy for the app

**Decision:** RLS is enabled on all tables. A single policy per table grants full access to the `authenticated` role (the one Supabase Auth user). `scripts/*.ts` use the **service role key**, which bypasses RLS entirely.

**Rationale:** Satisfies Supabase best-practice (RLS on by default) without per-row ownership complexity (AD-01 — no `user_id` to check). The service role key is appropriate for trusted server-side cron jobs that need unrestricted read/write across all tables (e.g. `notify.ts` joining `jobs`, `job_scores`, `notifications_log`).

**Alternatives considered:** RLS disabled entirely (rejected — Supabase/security best practice, and the dashboard's anon/authenticated key should not have unrestricted access by default even with one user); per-table custom policies beyond "authenticated = full access" (rejected — no ownership model exists to key policies on).

**Consequences:** The service role key must be kept out of any client-exposed code/env (`NEXT_PUBLIC_*`) — it lives only in GH Actions secrets, per `architecture.md` §5 dependency rules (only `infrastructure`/composition-root code touches it, and only in the `scripts/` entrypoint, never in `app/`).

### AD-13: `scrape_runs.status = 'partial'` descoped — `scrape.ts` writes only `success`/`failed` for now

**Decision:** `scripts/scrape.ts` records `scrape_runs.status` as `'success'` (adapter call completed without throwing) or `'failed'` (adapter threw) per source per run. The `'partial'` value stays in the `scrape_run_status` enum (no migration needed, AD-11 forward-only) but no code path writes it yet.

**Rationale:** `scrapers.md` §4 originally documented `'partial'` as "some companies/items failed but at least one succeeded (Greenhouse/Lever/Ashby only)." Producing it correctly requires `JobSourceScraper.fetchJobs` to report per-company success/failure counts back to `scripts/scrape.ts`, instead of the current per-company `console.warn`-and-continue inside each adapter. That's a `domain`-interface change (`JobSourceScraper.fetchJobs` return shape) touching all three ATS adapters plus `scrape.ts` — architect-level per `agent-profiles.md`, and out of scope for a no-scraper-changes/no-schema-changes merge-conditions pass.

**Alternatives considered:** Approve the `fetchJobs` signature change now (rejected — scraper/domain change, out of scope for this pass); drop `'partial'` from the enum (rejected — schema change requiring a migration, and AD-11 forward-only means it would persist on any historical rows regardless).

**Consequences:** `scrape_runs.status` is effectively binary (`success`/`failed`) until a future architect-approved `JobSourceScraper.fetchJobs` change adds per-company failure reporting. `docs/scrapers.md` §4 and `docs/database.md` note `'partial'` as reserved/not-currently-produced so the docs and schema don't silently diverge (`review-process.md` §4.2).

### AD-14: Lower keyword-gate default and make null `ai_score` rows retryable (supersedes AD-07/AD-08 null handling)

**Decision:** `KEYWORD_THRESHOLD` default lowered from `0.5` to `0.25` (still env-overridable). `SupabaseScoreRepository.insertScore` now upserts with `ignoreDuplicates: false`, so a conflicting `(job_id, role_selection_id)` row has `keyword_score`/`ai_score`/`ai_reasoning` updated rather than left untouched. `JobRepository.findUnscored` now returns jobs with no `job_scores` row **or** with an existing row where `ai_score IS NULL`, so stage 2 is retried on later `score.ts` runs.

**Rationale:** With the prior `0.5` default, real-world skill-overlap scores rarely cleared the gate, so stage 2 (OpenRouter) almost never ran and `ai_score` stayed `null` for nearly all jobs. Combined with `ignoreDuplicates: true` and `findUnscored` excluding any job with an existing `job_scores` row, this made every null `ai_score` permanent with no retry path — silently breaking the AI-scoring stage and, downstream, notifications (AD-08).

**Alternatives considered:** Raising AI usage by scoring every job regardless of keyword overlap (rejected — violates AD-07's "AI usage minimized" intent); adding a separate "retry" RPC/cron (rejected — unnecessary, `findUnscored`'s existing anti-join shape extends naturally to a `WHERE ai_score IS NULL OR no row` condition).

**Consequences:** More jobs reach stage 2 per run (bounded by role-title filter + `keyword_score >= 0.25`), a small, predictable increase in OpenRouter calls. Jobs whose AI call fails are retried indefinitely until they succeed or their `keyword_score` drops below the gate on a future re-run — no permanent-null dead end. AD-08's notification gate (`ai_score >= NOTIFY_THRESHOLD`) is unchanged but is no longer permanently blocked by a one-time AI failure. The GitHub Actions `vars.KEYWORD_THRESHOLD` repo variable, if set, still overrides this code default and must be lowered/unset for the change to take effect in CI.

### AD-15: Role-aware fetching — `JobSourceScraper.fetchJobs` takes `roles`, and `scrape.ts` skips when there's no active role selection

**Decision:** `JobSourceScraper.fetchJobs(companies: Company[], roles: readonly string[]): Promise<RawJob[]>` — every adapter (Greenhouse, Lever, Ashby, RemoteOK, Wellfound) now receives the active role selection's `expandedRoles` and filters its `RawJob[]` to those matching at least one role term in `title` or `description`, via the new shared pure helper `features/sources/domain/roleMatch.ts` (`jobMatchesRoles`). An empty `roles` array means "no filter" — adapters return everything fetched, preserving prior behavior as a safe default for any caller without a role selection. `scripts/scrape.ts` loads `SupabaseRoleRepository.getActiveSelection()` (the same call `score.ts` makes) before scraping; if there is no active role selection, it logs and returns without scraping or ingesting any source. This is the architect-approved `fetchJobs` signature change anticipated as out-of-scope in AD-13.

**Rationale:** Previously, role selection only filtered at scoring time (`findUnscored`'s title ILIKE match), so `scrape.ts` fetched and ingested every job from every configured source/company regardless of relevance — the vast majority of scraped/stored jobs were never relevant to the user's selected roles. Since no supported ATS API offers a role/keyword query parameter, the only correct fix is client-side filtering of the fetched `RawJob[]` before ingestion, using the same sanitization rules as the existing scoring-time ILIKE filter (`SupabaseJobRepository.sanitizeRoleForFilter`) so the two filters stay conceptually aligned. Skipping the entire scrape run when there's no active role selection avoids reintroducing the "ingest everything" problem during onboarding (before the user has made a role selection).

**Alternatives considered:** Keep filtering only at scoring time (rejected — doesn't address the root cause, jobs table fills with irrelevant rows); fall back to "fetch everything" when no active role selection exists (rejected — defeats the purpose for any run before role selection is configured, and silently re-ingests the noise this change removes); add a per-adapter ATS-side keyword query (rejected — none of Greenhouse/Lever/Ashby/RemoteOK/Wellfound's public APIs support one).

**Consequences:** `jobs` table growth is now bounded by the active role selection's `expandedRoles`, not by all configured companies/sources — switching the active role selection changes what future scrapes ingest (existing rows from a previous role selection are not retroactively removed). Any future caller of `fetchJobs` (tests, scripts, tooling) must pass a `roles` array; `[]` is the explicit "no filter" escape hatch and should be used deliberately, not as an oversight.

### AD-16: Cross-source duplicate detection via deterministic fingerprint, not fuzzy/AI matching (Phase 1 Task 1-3)

**Decision:** `jobs` gains `fingerprint` (`sha256(normalizeTitle(title) + "|" + normalizeCompanyName(companyName) + "|" + sortedLocationTags)`) and `canonical_company_name` (`normalizeCompanyName(companyName)`), both computed at write time in `SupabaseJobRepository` (not passed through `NormalizedJob`). `upsertMany` checks the fingerprint of every job not already known by `(source, source_job_id)` against all existing jobs; a match skips the insert and instead writes a provenance row to the new `job_duplicates` table (`canonical_job_id`, `source`, `source_job_id`, `url`), and refreshes the canonical job's `last_seen_at` so it isn't swept as expired while still listed elsewhere. `normalizeTitle` deliberately strips seniority tokens (senior/sr/junior/jr/lead/staff/principal/I-IV) so e.g. "Senior Backend Engineer" and "Backend Engineer" collapse to the same fingerprint; `normalizeCompanyName` strips trailing legal-entity (LLC/Inc/Corp/...) and regional-office (India/Singapore/...) suffixes.

**Rationale:** The only existing dedup was the DB-level `UNIQUE(source, source_job_id)` — it prevents the same source from re-inserting the same posting on every cron run, but does nothing when the same logical job is scraped from two different sources (e.g. Greenhouse directly and an aggregator), which produced duplicate rows, duplicate scoring runs, and duplicate Telegram notifications. Fingerprinting is deterministic and index-backed (no per-row fuzzy comparison, no AI call), matching the "avoid expensive comparisons, no AI" constraint. A plain (non-unique) index on `fingerprint` was chosen over a unique constraint so the app-level check-then-skip in `upsertMany` is the single source of truth for the "one row per logical job" invariant, rather than racing a DB constraint.

**Alternatives considered:** Fuzzy/Levenshtein title matching (rejected — expensive at scale, non-deterministic edge cases, explicitly out of scope per the task); AI-based semantic dedup (rejected — cost, non-determinism, explicitly excluded); a `UNIQUE(fingerprint)` DB constraint instead of app-level partitioning (rejected — would throw on any insert races or normalization drift instead of degrading to "treat as duplicate," and the pipeline is a sequential per-source cron, not a concurrent write path, so the extra DB-level guarantee isn't needed); deduping within a single scrape batch by fingerprint in addition to the cross-run DB check (deferred — two same-source postings sharing a fingerprint in one batch is rare and not the case this task targets; documented as a known limitation, not implemented, to avoid the added complexity of resolving IDs for rows not yet inserted).

**Consequences:** A job discovered second (from a different source) never gets its own `jobs` row, `job_scores` row, or Telegram notification — `job_duplicates` is the only record of it, so any future UI wanting to show "also listed on N other sources" reads from there. Existing rows ingested before this migration have `fingerprint = ''` until `npm run backfill:fingerprints` is run once; until then they're invisible to the cross-source check (never falsely merged, just not yet deduped against). `UpsertResult` gained a required `duplicates` count, and `scrape_runs` gained `duplicate_count` — both existing call sites were updated in the same commit (no silent breakage). Seniority-token stripping in `normalizeTitle` is a deliberate tradeoff: a "Senior" and non-senior posting for the same title/company/location will now be treated as one logical job even though the levels may differ; acceptable per this task's explicit examples, revisit if it causes real false merges.

### AD-17: Notification pipeline verified exactly-once/retry-safe; digest mark-as-notified batched (Phase 1 Task 4)

**Decision:** Audited the full notification pipeline (`sendNotification`, `sendDigest`, `sendDigestMvp`, `SupabaseNotificationRepository`, `TelegramBotSender`) against the "exactly once, retry safe, idempotent, duplicate safe" requirement. Confirmed: `notifications_log` has `UNIQUE(job_id)` and every write goes through `on conflict (job_id) do nothing`, so re-running the same match twice never inserts a second row or double-counts. `TelegramBotSender.post()` throws on any non-2xx or `{ok:false}` response (including the recent `BUTTON_URL_INVALID` case), and every send site only calls `markNotified`/`markManyNotified` *after* a successful send, inside code paths that leave the job(s) unmarked on failure -- so a failed send is correctly retried on the next `score.ts`/`notify.ts` cron run rather than silently dropped or falsely marked sent. One real gap found and fixed: `sendDigest`/`sendDigestMvp` send ONE Telegram message covering many matches, but marked each job notified in a per-item loop calling `markNotified` — if a write N matches into the loop failed, matches before it were marked (correct) but matches after it were not, even though all of them were already shown in the one message that was sent, contradicting the code's own "all jobs unmarked for retry" comment. Added `NotificationRepository.markManyNotified(jobIds)` (single upsert, `on conflict (job_id) do nothing`, no-op on empty array) and switched both digest senders to call it once instead of looping.

**Rationale:** `sendNotification` (individual mode) is correctly one-send-per-job with one-mark-per-job, so a per-match loop there is appropriate — each match is independent. The digest senders send one message for N jobs, so the write that follows should also be one atomic-ish batch operation matching that unit of work, not N independent ones with an inconsistent partial-failure window.

**Alternatives considered:** Wrap the digest send+mark in a DB transaction (rejected — the Telegram HTTP call can't participate in a Postgres transaction anyway, and a batched upsert already shrinks the partial-failure window from N round trips to 1, which is enough for a personal-scale tool); leave the per-item loop as documented behavior (rejected — the code comment already claimed all-or-nothing semantics that weren't actually true, a real correctness/documentation mismatch worth closing).

**Consequences:** `NotificationRepository` gained a required `markManyNotified` method — all three test fakes (`sendNotification.test.ts`, `sendDigest.test.ts`, `sendDigestMvp.test.ts`) and `SupabaseNotificationRepository` were updated in the same commit. Residual known gap (documented, not fixed — out of scope for a personal tool): if the Telegram send succeeds but the immediately-following `markNotified`/`markManyNotified` call itself throws (e.g. a transient DB error), the job(s) will be re-sent on the next run since `notifications_log` was never written — an at-least-once, not exactly-once, guarantee in that narrow window. Fixing this fully would require an outbox/transactional-write pattern, disproportionate to the risk for a single-user tool with infrequent DB write failures.

### AD-18: Source-level health summary computed from `scrape_runs`, independent of `companies.health_status` (Phase 1 Task 5/7)

**Decision:** Added `scrape_runs.failure_category` (plain `text`, values from the new `FailureCategory` TS union in `classifyScrapeFailure.ts`: `timeout | parsing | selector | captcha | blocked | authentication | rate_limited | not_found | empty_feed | unknown`), set by `scripts/scrape.ts` on every run -- via `classifyScrapeFailure(err)` when the adapter throws, or `'empty_feed'` when it returns zero raw jobs on an otherwise-successful run. A new pure function `computeSourceHealthSummary(source, runs)` aggregates a source's recent `scrape_runs` into `{ successRate, avgLatencyMs, consecutiveFailures, lastSuccessAt, lastFailureAt, recoveryDetected, topFailureCategory, recommendation }`; `getSourceHealthReport()` runs it for every `JOB_SOURCES` entry via the new `ScrapeRunRepository.listRecentBySource(source, limit)`. `recommendation` is deterministic rule-based text (no AI), thresholded against the existing `SOURCE_HEALTH_CONFIG.disableAfterConsecutiveFailures`.

**Rationale:** `companies.health_status`/`consecutive_failures` (AD from `20260619000010_source_health.sql`) only exists for board-token sources (greenhouse/lever/ashby) because `companies` rows only exist for those -- wellfound/remoteok/mycareersfuture have no row to hold health state at all, so they had zero health visibility previously. Separately, `companies.health_status` is only updated by the standalone `validate-sources.ts` probe cron, never by `scrape.ts` itself (AD-13 noted this as a known gap: a company whose actual scrape consistently fails is invisible until the next probe run, which checks board reachability, not the adapter's actual parse/fetch success). Computing health from `scrape_runs` -- already written by every source on every cron run regardless of company config -- closes both gaps without touching the `companies` schema or the existing, working probe-based auto-disable flow.

**Alternatives considered:** Extend `companies.health_status` tracking to also update from `scrape.ts` failures (rejected for this pass -- requires changing `JobSourceScraper.fetchJobs`'s per-company error-swallowing to report failures back to the caller, an architect-level interface change across all 5 adapters per AD-13, and doesn't solve the feed-based-source gap since those sources have no `companies` row regardless); a full time-series metrics table (rejected -- `scrape_runs` already has everything needed: status, timing, error; a new table would duplicate data Caveman-style rather than reuse it); Postgres enum for `failure_category` (rejected -- an enum-alter migration is needed for every new category; plain `text` with the fixed set living in TypeScript is simpler to extend, consistent with `metadata`/`error` already being unconstrained at the DB level).

**Consequences:** This is a second, independent health signal alongside `companies.health_status` -- they are not merged or reconciled, and `scrape.ts`'s scraper-selection logic (`listActiveHealthy`) still only reads `companies.health_status`, so this new summary does not yet drive auto-disable/auto-skip behavior; it is observability-only as of Phase 1 (no UI wired -- CLAUDE.md's domain/application/infrastructure-before-UI rule -- Phase 4 analytics is the intended consumer). `selector`/`captcha` failure categories are unreachable by any current adapter (none does HTML/DOM scraping or hits a CAPTCHA wall) -- deliberate extension points, not dead code to be removed.

### AD-19: `job_scores.retry_count` via an atomic `upsert_job_score` RPC, plus AI-retry queue monitoring (Phase 1 Task 6)

**Decision:** Added `job_scores.retry_count integer not null default 0`, incremented only via a new `upsert_job_score(...)` Postgres function that does `INSERT ... ON CONFLICT (job_id, role_selection_id, resume_version) DO UPDATE SET ..., retry_count = job_scores.retry_count + (1 if excluded.ai_score IS NULL else 0)` in one round trip -- mirroring the existing `set_active_resume`/`set_active_role_selection` RPC pattern (AD-09) rather than a plain `.upsert()`. `SupabaseScoreRepository.insertScore` now calls this RPC instead of `.from("job_scores").upsert(...)`. Added `ScoreRepository.findAwaitingAi(roleSelectionId, resumeVersion, keywordThreshold)` (job_scores rows past the keyword gate with `ai_score IS NULL`, ordered oldest `scored_at` first), a pure `computeScoringQueueSummary(awaiting, stuckThresholdHours)` (awaiting count, oldest-pending age, stuck-job list, max/avg retry count), and `getScoringQueueReport()` composing the two. `scripts/score.ts` logs the summary after every run and warns with stuck job ids.

**Rationale:** `job_scores.scored_at` is set once at first insert and never touched again by a plain upsert (the column isn't in the client-supplied payload, so `ON CONFLICT DO UPDATE` never rewrites it) -- that makes it a stable, correct signal for "how long has this job been in the queue" (oldest-pending / stuck detection), but it says nothing about "how many times has scoring been attempted." That second signal needs its own counter, and incrementing it conditionally ("only when this write still leaves `ai_score` null") isn't expressible through a plain client-side `.upsert()` payload -- it requires either a read-modify-write per job (an extra round trip per job during every `score.ts` run) or a small SQL function that does it atomically in the same statement as the write already happening. The RPC was chosen over the read-modify-write for exactly the reasons AD-09's swap functions already established: one round trip, no races.

**Alternatives considered:** Approximate retry count from `(now - scored_at) / cronIntervalHours` (rejected -- fragile, assumes a fixed cron cadence, and CLAUDE.md explicitly says avoid magic); a read-modify-write in `scoreJob.ts`/`score.ts` instead of an RPC (rejected -- doubles DB round trips for every job in the scoring loop, for a personal-scale tool where the RPC costs nothing extra); a separate `job_score_attempts` history table (rejected -- overkill for a single counter, Caveman-style over-engineering for what Task 6 actually asks for).

**Consequences:** `ScoreRepository.insertScore`'s implementation changed (plain upsert → RPC) but its interface/behavior contract is unchanged for callers -- `scoreJob.ts` is untouched. `SupabaseScoreRepository.test.ts` was updated to assert on `client.rpc(...)` instead of `builder.upsert(...)`. The queue/stuck-job report is logged, not enforced -- no auto-drop, no notification, no UI (Phase 4 territory); AD-14's indefinite retry remains the actual "never permanently stuck" mechanism, this just makes long waits visible so an operator can investigate.

### AD-21: Contact email extraction is plain-text/regex only; structured-HTML and mailto: are deferred (Phase 2 Task 9)

**Decision:** Replaced the existing, unused `extractRecruiterEmail` (`shared/infrastructure/text.ts` -- defined, tested, never called from any code path) with `extractContactEmail` (`features/jobs/domain/extractContactEmail.ts`), which extracts every email in `title+description`, excludes fully-automated mailboxes (noreply/unsubscribe/postmaster/...), categorizes the rest by a local-part keyword heuristic into `recruiter > hr > hiring_manager > company_contact` (task's stated priority order) with a `high|medium|low` confidence, and returns the single highest-priority match. Wired into `ingestJobs.ts` alongside the existing `parseMinYears` call, storing `jobs.contact_email`/`contact_email_category`/`contact_email_confidence`.

**Rationale:** The task's preferred methods are "structured HTML, regex, mailto links" -- but by the time a job reaches `ingestJobs`, its `description` has already been through `stripHtml()` inside each scraper adapter (scrapers.md §3, applied before the `RawJob` is even constructed), so `<a href="mailto:...">` markup and any HTML structure are already gone. Regex over the remaining plain text is the only one of the three methods actually available without changing what every scraper adapter passes through the pipeline (an architect-level, cross-adapter interface change, same category of change AD-13/AD-18/AD-20 already deferred). The categorization heuristic operates on the email's local part only (not surrounding-text proximity/NLP) to stay within "no AI"/"avoid magic" -- a personal-name address with no recognizable keyword honestly reports `company_contact`/`low` rather than guessing "hiring manager" from context.

**Alternatives considered:** Extract from raw HTML before `stripHtml()` runs, requiring every scraper adapter to also pass the pre-strip HTML through to `ingestJobs` (rejected -- cross-cutting interface change across 5 adapters, same category of risk as the deferred `fetchJobs` change in AD-13/18); proximity-based text search for phrases like "hiring manager" near a personal-name email (rejected -- heuristic complexity for a marginal accuracy gain, and starts to resemble the "magic" CLAUDE.md says to avoid); keep the old `extractRecruiterEmail` alongside the new function (rejected -- CLAUDE.md forbids duplicated logic, and the old function's single-tier "first non-excluded email" behavior is strictly subsumed by the new categorized/prioritized one).

**Consequences:** Emails only reachable via a `mailto:` href with non-email link text (e.g. `<a href="mailto:jane@co.com">Apply now</a>`) are invisible to this extraction -- the address never appears in the stripped plain text. This is a real, documented gap (`design/limitations.md`), not a silent one. `contact_email_category`/`confidence` are plain `text` columns (not Postgres enums), consistent with `scrape_runs.failure_category`'s established convention in this codebase.

### AD-22: Salary extraction via layered regex patterns, requiring a currency/unit/period signal to avoid false positives (Phase 2 Task 10)

**Decision:** `extractSalary` (`features/jobs/domain/extractSalary.ts`) parses `jobs.salary_currency/salary_min/salary_max/salary_period/salary_confidence` from `title+description` at ingest, alongside the existing `parseMinYears`/`extractContactEmail` calls. Three regex patterns are tried in priority order -- currency-symbol-first (`₹18-24 LPA`, `$120k/year`, `Rs. 50,000/month`), number-then-currency-code-or-LPA/lakh (`20 LPA`, `35 USD/hour`, `8-10 lakhs per annum`), and number-with-explicit-period-but-no-currency (`5000-7000 per month`) -- each requiring at least one of a currency symbol/code, an LPA/lakh unit, or an explicit period phrase directly attached to the number(s). A bare number with none of these (e.g. "5+ years of experience") is never treated as a salary. `LPA`/`lakh` doubles as both a currency signal (implies INR) and a magnitude multiplier (1 lakh = 100,000) and an implicit "yearly" period (the "PA" in LPA is literally "Per Annum") when no more specific period word is also present. `confidence` is `'high'` when both currency and period were determined, `'medium'` when only one was. Explicit "Negotiable"/"Competitive"/"DOE" text with no figure returns an object with all-null numeric fields but `confidence: 'low'` (a salary section exists, just no number) -- distinct from returning `null` outright when there's no salary-related text at all.

**Rationale:** Salary formats in real job postings vary enough (currency symbol vs. code vs. India-specific LPA/lakh units; range vs. single figure; period as a suffix, a `per X` phrase, or implied) that a single regex covering every case correctly would be unreadable and hard to verify; three prioritized patterns, each independently testable, is the deterministic middle ground between "one giant unmaintainable regex" and "a parser generator/grammar" (avoids overengineering in the other direction). Requiring a signal (currency/unit/period) before accepting a numeric match is the key false-positive guard -- job descriptions are full of bare numbers (years of experience, team size, percentages) that must never be misread as a salary figure.

**Alternatives considered:** A general-purpose NLP/currency-parsing library (rejected -- new dependency, "avoid unnecessary dependencies," and this codebase already established the local-regex-plus-tests pattern for `parseMinYears`/`extractContactEmail`); defaulting an unspecified period to "yearly" for any bare `$Nk` figure (rejected -- ambiguous in general, could silently misrepresent an hourly/monthly figure as annual; only inferred for LPA where the annum meaning is unambiguous by definition of the unit itself); AI-based extraction (rejected, explicitly out of scope per the task).

**Consequences:** Formats not matching any of the three patterns (e.g. spelled-out numbers, unusual currency abbreviations, salary given only as a benefits-page link) are not extracted -- `extractSalary` returns `null`, same as "no salary mentioned," which is a real (documented) false-negative risk rather than a false-positive one, consistent with this codebase's general "fail safe, not silently wrong" bias (mirrors AD-16's fingerprint-dedup and AD-20's career-page-discovery choices). `salary_min`/`salary_max` are stored as `numeric` rather than `integer`, matching `job_scores.estimated_cost_usd`'s existing precedent for money-like values in this schema.

**Decision:** `OpenRouterAiScoreProvider`'s prompt-building now truncates `resume.parsedText` and `job.description` via a new `truncateText(text, maxChars)` helper before embedding them in the AI request -- `OPENROUTER_MAX_RESUME_PROMPT_CHARS` (default 4000) and `OPENROUTER_MAX_DESCRIPTION_PROMPT_CHARS` (default 2000), both env-overridable, both read per-call (not module-level) so a config change takes effect immediately. ### AD-26: Composite ranking score is a persisted, deterministic additive blend of ai_score, computed once at scoring time -- not a read-time recomputation, not ML (2026-07-04 continuous-improvement pass, Theme 1)

**Decision:** `job_scores.overall_score` (nullable, mirrors `ai_score`'s NULLS-LAST semantics) is set by `computeOverallScore(job, aiScore, preferences)` inside `scoreJob.ts` whenever `aiScore` is non-null: `overall_score = min(1, ai_score + companyBonus? + remoteBonus? + salaryBonus?)`, gated on `RankingPreferences` (`app_settings` key `ranking_preferences`, same JSON-blob pattern as `notification_preferences`) -- preferred-company substring match, `preferRemote` + `locationTags` includes `remote`, and salary-disclosed (either `salaryMin`/`salaryMax` non-null). Reasons applied are stored alongside as `overall_score_reasons` for dashboard display. The dashboard's default sort key moved from `ai_score` to `overall_score`, with `posted_at desc` retained as the tiebreaker. `upsert_job_score` gained `p_overall_score`/`p_overall_score_reasons` as trailing parameters with `default null` (migration `20260704000004_ranking_overall_score.sql`, renumbered from `...000003` to avoid colliding with AD-25's `20260704000003_job_attributes.sql`, an independent same-day migration from a parallel session), which `CREATE OR REPLACE FUNCTION` allows without creating a duplicate overload. Pre-existing rows are backfilled in the same migration (`overall_score = ai_score` where currently null) so they don't sink under the new sort just for predating the column.

**Rationale:** The insights feature already established a "compute at read time, don't persist" precedent for skill gaps (`docs/plans/phase-p1-insights.md`), but that doesn't transfer cleanly here: the dashboard's pagination is genuine server-side `LIMIT`/`OFFSET` via PostgREST, and re-sorting by a value only known after fetching a page would either require fetching every matching row per page load (a real, riskier change to a core, heavily-tested query path with no live Supabase available this session to validate against) or would only reorder within a page, silently failing to bubble a page-2 preferred-company job onto page 1. Persisting `overall_score` at scoring time keeps the existing pagination/sort architecture completely intact -- swap one `ORDER BY` column for another, same nullability contract. A plain additive bonus model (not a normalized weighted-average) keeps the score trivially explainable ("this job got +5% because it's a preferred company") and keeps `aiScore` as the dominant term by construction, rather than requiring a weight-redistribution scheme where bonuses could unintentionally swamp the AI signal.

**Alternatives considered:** Recomputing overall score at dashboard-read time from `JobWithScore` (rejected -- see pagination conflict above; also `JobWithScore` currently omits `canonicalCompanyName`/`salaryMin`/`salaryMax` from the dashboard query, so this would require widening that select too); extending `upsert_job_score`'s existing parameters in place instead of appending new ones (rejected -- would break the documented "same signature" requirement for `CREATE OR REPLACE FUNCTION` and risk an orphaned overload); a full weighted-average composite score with freshness as an explicit weighted term (rejected -- freshness is already handled by the existing `posted_at desc` tiebreaker, so adding it as a bonus too would double-count it); an ML/embeddings-based relevance re-ranker (explicitly out of scope per this session's constraints -- "avoid ML, embeddings, or complex recommendation engines").

**Consequences:** A job's `overall_score` reflects the `RankingPreferences` in effect *at the time it was last (re-)scored* -- changing preferences doesn't retroactively re-rank already-scored jobs until `score.ts` next processes them (it only ever processes unscored/retry-queue jobs, per the existing scoring pipeline), so a preference change's effect appears gradually as new jobs are scraped and scored, not immediately across the whole dataset. This mirrors the existing, already-accepted "scores don't retroactively update on resume change" limitation (`design/limitations.md`) rather than introducing a new class of staleness.

### AD-28: Verification framework severity/diagnostics refinement -- per-outcome severity overrides, structured diagnostics fields, and one closed coverage gap (v1.x operational-excellence pass)

**Decision:** Three additive changes to the framework built in AD-27, none of which change its architecture: (1) `CheckOutcome` gained an optional `severityOverride` field -- resolved by `runChecks()` into `CheckResult.severity` as `severityOverride ?? check.severity` -- so a single outcome can be scored at a different severity than the check's typical failure mode; (2) `CheckOutcome` gained `probableCause`/`suggestedFix`/`affectedSubsystem`/`docReference` (replacing the single ad hoc `recommendation` string), populated on every non-pass branch across all 25 pre-existing checks, and rendered by both the console and Markdown reporters; (3) a new external check, `external.telegram-webhook`, closes a real, previously-zero-coverage gap -- the Telegram webhook (`src/app/api/telegram/webhook/route.ts`, registered by `npm run setup:webhook`) that drives digest-mode "Worth Reviewing" pagination had no verification at all. A new shared `skipOutcomes.ts` (`SKIPPED_NO_SUPABASE_CLIENT`, `skippedMissingCredential()`) consolidates ~15 near-duplicate "client unavailable" branches into one low-severity outcome. `computeHealthScore`'s `recommendations` list now deduplicates identical `suggestedFix` text. `scripts/verify-production.ts` was changed so the source-health/stale-sources/scoring-queue checks always appear in the report (as a consistent skip) instead of being conditionally omitted when no Supabase client exists. Separately, `src/shared/infrastructure/envCheck.ts`/`connectivityCheck.ts` (and `scripts/doctor.ts`, which shares them) had their status vocabulary unified from `"warn"` to `"warning"` to match the verification framework's `CheckStatus` exactly.

**Rationale:** A review pass (this session's Phase 1) found that severity was being applied uniformly per-*check* rather than per-*outcome*: e.g. `infra.env-vars` is critical because a missing *required* var is critical, but its own "relying on a documented default" warning branch inherited that same critical severity, over-penalizing a benign, expected state. The more consequential version of the same bug: when Supabase credentials are absent, ~15 independent checks each reported "can't query" at their own (often high/critical) severity for the exact same root cause, which the pre-refinement health score algorithm summed -- one missing env var could floor the score at 0 by being counted a dozen times over, obscuring whether there was one problem or many. `severityOverride` fixes this without changing the check/runner contract: a check still reports one severity by default, but can explicitly downgrade a specific outcome when it knows that outcome is a symptom, not a new finding. The structured diagnostics fields directly satisfy this session's mission ("each failed or warning check should provide probable cause, suggested fix, affected subsystem, severity, documentation reference") and were verified to make a real difference by running the framework end-to-end before and after: the aggregate health score moved from a floored 0/100 (indistinguishable from "everything is broken") to a representative 44/100 ("one critical problem, many downstream skips") for the identical no-credentials scenario.

**Alternatives considered:** Leaving severity purely per-check and instead deduplicating at the health-score level by root-cause detection (rejected -- would require each check to expose a "root cause ID" for the score aggregator to group by, a bigger structural change than an optional per-outcome override, for the same practical result); renaming the `recommendation` field to `suggestedFix` without adding the other three diagnostic fields (rejected -- the mission explicitly asked for probable cause/affected subsystem/doc reference too, and a single free-text field was already shown, in the initial AD-27 build, to produce vague "review X" messages rather than actionable ones); a standalone `digest_sessions` activity check to accompany the new webhook check (rejected -- see `docs/operations/production-verification.md` §4, real false-positive risk on fresh installs and non-digest modes); renaming `npm run diagnose`/`npm run diagnostics` to remove the naming collision (rejected -- both names are already documented and used across multiple docs; the lower-risk fix is a clear disambiguating note wherever they're listed together, not a disruptive rename of an established command for cosmetic clarity).

**Consequences:** Every one of the 25 pre-existing checks was touched to add the new diagnostic fields, and one new check (`external.telegram-webhook`) brought the total to 26 and, where applicable, a `severityOverride` -- a mechanical but repo-wide change, verified by re-running `tsc --noEmit`, the full test suite (two new tests added: severity-override resolution in `runChecks.test.ts`, recommendation deduplication in `computeHealthScore.test.ts`), `check:service-role-boundary` (caught and fixed two reintroduced literal occurrences of the service-role key's env var name in prose diagnostic strings -- the boundary gate does not distinguish code from string literals, by design), and an end-to-end no-credentials run of `scripts/verify-production.ts` confirming the deduplicated-recommendations and non-floored-score behavior directly. `TECHNICAL_DEBT.md` #1 (the `scrape.yml` schedule doc/reality mismatch) is referenced, not re-litigated, by `infra.scheduler-config`'s updated comment -- this pass corrected the check's own stale assumption ("commented out pending go-live") to match the already-established fact that the schedule is live, without touching the still-open human-decision question that debt item tracks.

### AD-30: Resume module gains DOCX support and a sha256 parse-once cache; existing `resume` feature module extended in place, not replaced (merge-workspace Phase 1)

**Decision:** `resumes.content_hash` (nullable `text`, non-unique index) was added, and `set_active_resume` gained a fourth parameter, `p_content_hash`. `application/uploadResume.ts` now takes a `parseText: (buffer, mimeType) => Promise<string>` port instead of pre-parsed text, and looks up `ResumeRepository.findByContentHash(contentHash)` before calling it — on a hit, `parseText` is never invoked and the cached row's `parsedText` is reused verbatim; on a miss, `parseText` runs and the result is validated by a new `domain/validateParsedText` (rejects empty/near-empty text, e.g. a scanned PDF) before being stored. `infrastructure/parseDocx.ts` (mammoth's `extractRawText`) was added alongside the existing `parsePdf.ts`, dispatched by MIME type via `infrastructure/parseResumeFile.ts`. `actions.ts` computes the sha256 up front and uses it as the Storage path (`<hash>.pdf` / `<hash>.docx`), so identical re-uploads overwrite the same Storage object too. The existing `src/features/resume/` module (singular) was extended in place rather than creating a separate `src/features/resumes/` module, despite the merge plan's original phrasing — a second, near-duplicate feature module for the same table/RPC/UI would itself violate CLAUDE.md's "no duplicated types" rule.

**Rationale:** jobhunt-app's Python reference (`jobhunt/extract.py`) hashes file bytes with `hashlib.sha256` specifically as a "never re-parse this content again" cache key (its docstring says so directly) — the same shape carries over cleanly to a Postgres column + index rather than a new subsystem. jobhunt bug #6 is `_from_docx` iterating only `document.paragraphs`, which python-docx does not populate with table cell text; mammoth's `raw-text.js` recurses generically into every element's `children` regardless of type (`w:tbl`/`w:tr`/`w:tc` included), so the equivalent bug does not reproduce here — verified directly against `node_modules/mammoth/lib/raw-text.js` (not just asserted) and by a test building a real in-memory `.docx` fixture with `jszip` and asserting table-cell text survives extraction, per this project's Phase 0 lesson that a design-gating finding must be checked against the actual file, not taken on faith. The empty-text check is a domain rule, not infra-specific, so both PDF and DOCX share one `validateParsedText` call site instead of duplicating the check per parser.

**Alternatives considered:** A separate `src/features/resumes/` (plural) module as literally specified in the merge plan (rejected — the table, RPC, Storage bucket, and UI all already exist under the singular `resume` module; a second module would either duplicate `ResumeRepository`/`Resume` or import across feature boundaries, both worse than the plan's literal wording); reactivating an existing version row on a cache hit instead of always inserting a new version (rejected — `job_scores.resume_version` keys scoring history off distinct version numbers, and every other upload path already always creates a new version; skipping that only for hash-matched re-uploads would be a special case with no clear benefit, since the cache's entire purpose is skipping the parse, not the versioning); a unique index on `content_hash` (rejected — multiple version rows are expected to share a hash whenever a user re-uploads the same file, so uniqueness would break the very re-upload path the cache exists for); building LLM-based resume parsing now (explicitly out of scope per the merge plan — deferred to Phase 3's `AiScoreProvider`/OpenRouter decision, no stub interface was needed since nothing in this phase calls an LLM).

### AD-31: Local embedding similarity is a new, informational stage-2 signal (`job_scores.embedding_score`) — not a stage-1 gate, not folded into `overall_score`'s ranking (merge-workspace Phase 2)

**Decision:** A new `EmbeddingScoreProvider` port (domain, mirrors `AiScoreProvider`'s shape: `score({job, resume}) => Promise<number | null>`, never throws) is implemented by `TransformersEmbeddingScoreProvider` (infrastructure), using `@huggingface/transformers`' `feature-extraction` pipeline (`onnx-community/all-MiniLM-L6-v2-ONNX`, mean-pooled + normalized) run **locally, offline, at zero per-call cost** — no API key, no network call at inference time beyond the one-time model weights download (cached on disk after). `scoreJob.ts` calls it under the *same* `keywordThreshold` gate as the existing AI call (i.e. it is a stage-2 concern, computed alongside `aiScore`, not a replacement for stage 1's keyword gate). The result is stored as a new nullable `job_scores.embedding_score` column (migration `20260714000001_job_scores_embedding_score.sql`, `upsert_job_score` gains `p_embedding_score` as a trailing default-null parameter, same `CREATE OR REPLACE FUNCTION` pattern as AD-26). It is **not** wired into `computeOverallScore`'s ranking blend or the dashboard sort — `overall_score` continues to be exactly what AD-26 specified (ai_score + deterministic bonuses). `embedding_score` is persisted for visibility/future use only this phase.

**Rationale:** This directly ports jobhunt's hybrid scoring (`jobhunt/scoring.py`: `W_SEM*semantic + W_SKILL*overlap`, mirrored here as a same-shape resume-vs-job cosine similarity) per the merge plan's Phase 2 scope, and fixes two concrete jobhunt bugs found in the reference implementation while porting it: bug #1 — `scoring.py`'s `(sim + 1) / 2 if sim < 0 else sim` only remaps *negative* similarities, leaving positive ones passed through raw, which is a discontinuity at `sim=0` (a job at `sim=0.01` scores ~0.01 while one at `sim=-0.01` scores ~0.495); `cosineSimilarityToScore()` here applies `(sim + 1) / 2` to every value unconditionally, clamped to `[0,1]`, so the mapping is continuous everywhere. Bug #7 — jobhunt's `_semantic()` swallows any failure (`except Exception: return None`) with no log line, so a broken/uninstalled model silently degrades every score to overlap-only with no operator-visible signal; `TransformersEmbeddingScoreProvider` logs a `console.warn` on every null-returning path (empty resume/job text, or a caught exception), matching `OpenRouterAiScoreProvider`'s existing failure-logging convention for the same reason (the pattern already established for stage-2 AI failures) so a degraded run is diagnosable from `score.ts` output, not just inferred from a null column. This reconciles with two standing decisions that look like blanket rejections of embeddings but were scoped narrower than they read in isolation: AD-07 rejected embeddings *as the stage-1 gate*, specifically because that would "add an embeddings API dependency/cost" — a concern that assumed a paid embeddings API and does not apply to a model that runs locally with zero marginal cost per call; AD-26 rejected "an ML/embeddings-based relevance *re-ranker*" as a replacement for the deterministic composite bonus formula, which this isn't — `embedding_score` doesn't touch `overall_score` or dashboard ordering at all. `ROADMAP.md`'s v2-candidates entry ("worth revisiting only if skills-dictionary maintenance becomes a real, recurring burden, not preemptively") is superseded for this specific, narrower shape by the merge plan's explicit Phase 2 scope (a standing, user-approved cross-repo plan, not a preemptive addition invented this session) and updated accordingly.

**Alternatives considered:** Using embedding similarity to replace or gate the stage-1 keyword filter (rejected — this is exactly what AD-07 rejected, and re-opening it wasn't in Phase 2's scope); blending `embedding_score` into `overall_score`'s ranking formula (rejected — would reopen AD-26's explicit "not ML" ranking-composite promise and its "aiScore is the dominant term by construction" design goal for no requirement in the merge plan's Phase 2 checklist, which only asks that embedding similarity be computed and added to stage-2 scoring, not that it drive ranking); `@xenova/transformers` (the older, deprecated-successor package name commonly associated with "Transformers.js") instead of `@huggingface/transformers` (rejected — the latter is the actively maintained v3+/v4 successor and is what `npm view` shows as the current, non-deprecated package); gating the embedding call behind its own on/off env var like jobhunt's `use_embeddings` sidebar toggle (rejected — out of scope for Phase 2's checklist, and the provider already degrades gracefully to null-with-a-logged-warning on any failure, so an explicit toggle adds a knob without a concrete need for one yet).

**Consequences:** `score.ts` cron runs now do one extra local model call per job that clears the keyword gate; the model (~90 MB) downloads once on first invocation per machine/container and is cached afterward by `@huggingface/transformers`, so cold CI/serverless environments without a persistent cache would re-download it each run — acceptable for this project's cron-on-a-persistent-runner deployment shape, revisit only if that deployment shape changes. `npm run verify`'s test suite never invokes the real pipeline (the `feature-extraction` pipeline is mocked in `TransformersEmbeddingScoreProvider.test.ts`; the continuous-mapping and cosine-similarity math is tested directly as pure functions in `embeddingSimilarity.test.ts`), so tests stay fast and deterministic without a model download. `embedding_score` has no dashboard surface yet — a future phase that wants to show or rank on it needs its own scoped pass, not assumed here.

**Consequences:** `Resume`/`NewResume` gained a `contentHash` field (`string | null` on read, required on write), which is a breaking shape change absorbed by the two other call sites that construct `Resume` fixtures directly (`scoreJob.test.ts`, `OpenRouterAiScoreProvider.test.ts`). `mammoth` was added as a runtime dependency and `jszip` as a dev-only dependency (test-fixture construction only; already a transitive dependency of `mammoth`, listed explicitly rather than relied on implicitly). `design/erd.md`, `tech-stack.md`, `limitations.md`, `scope.md`, `security.md`, `api-reference.md`, `user-guide.md`, and `use-cases.md` were all updated per CLAUDE.md's document-maintenance table; the parallel legacy `docs/database.md` etc. tree was left as-is, consistent with Phase 0's precedent of only maintaining `design/` going forward (tracked as pre-existing drift in `TECHNICAL_DEBT.md` #1).

### AD-32: A second, separate LLM abstraction (`src/shared/infrastructure/llmClient.ts`) for resume suggestions — not a replacement for `AiScoreProvider`/OpenRouter, and REST calls rather than provider SDKs (merge-workspace Phase 3)

**Decision:** The merge plan's Phase 3 line item "`src/lib/llm/` abstraction, env-var provider switch, Gemini Flash default, Anthropic optional" is implemented as `src/shared/infrastructure/llmClient.ts` (not a new top-level `src/lib/llm/` directory) exporting `completeLlm({system, user, maxTokens, jsonMode}) -> {text, provider, model}`, switching on `LLM_PROVIDER` (`gemini` default, `anthropic` optional) with `LLM_MODEL` as an override, mirroring jobhunt/llm.py's `complete()` shape. It talks to Gemini's and Anthropic's REST APIs directly via `fetchWithRetry` (`src/shared/infrastructure/http.ts`) — the same pattern `openrouterClient.ts` already uses for OpenRouter — rather than installing `@google/genai`/`@anthropic-ai/sdk`. It is a **new, second** AI-client abstraction alongside `openrouterClient.ts`/`AiScoreProvider`, not a replacement: `AiScoreProvider` (scoring.md §3) continues to back job-vs-resume scoring through OpenRouter's multi-model gateway, untouched. `llmClient.ts` backs only the new `ResumeSuggestionProvider` port (`src/features/resume/domain/`) used by the new resume-suggestions feature.

**Rationale:** Phase 0's findings log flagged this exact fork in the road ("extend/replace the `AiScoreProvider` port vs. add a new `src/lib/llm/` layer alongside it"). Replacing `AiScoreProvider` was rejected because it is a working, tested, differently-shaped abstraction (schema-constrained JSON scoring through a multi-model gateway) that Phase 2's `EmbeddingScoreProvider`/AD-31 already builds on top of — swapping its transport would be a large, unrequested blast radius for a phase whose actual ask (resume coaching text generation, not job scoring) doesn't touch it at all. `src/lib/` was rejected as the location because it doesn't exist as a business-logic convention in this codebase — the only existing file there is shadcn's `cn()` helper (`src/lib/utils.ts`), a framework/tooling convention, not this project's pattern for cross-feature infrastructure; `design/architecture.md` §10 already documents `src/shared/infrastructure/` as that location (`openrouterClient.ts`, `http.ts`, etc.), so the new client was placed there to stay consistent with the codebase's actual, documented layering rather than the plan's literal wording. Direct REST over an SDK matches `openrouterClient.ts`'s own precedent exactly (`fetchWithRetry`, `requireEnv`/`optionalEnv`, a typed `*Error` with a `reason` classification) — introducing two new heavy provider SDKs when a REST call already works, is already the established pattern, and avoids two more `npm audit` surfaces would be inconsistent with no benefit.

**Alternatives considered:** Replacing `AiScoreProvider`'s OpenRouter backend with a provider-switchable client so scoring and resume suggestions share one abstraction (rejected — see Rationale; also OpenRouter's schema-constrained JSON mode has no equivalent on Anthropic's REST API, so unifying would mean either losing that guarantee for scoring or building two code paths inside one port, worse than two small, honestly-separate ports); `src/lib/llm/` as the plan literally states (rejected — not this codebase's convention, see Rationale); `@google/genai` + `@anthropic-ai/sdk` official SDKs (rejected — `openrouterClient.ts` proves direct REST is sufficient and is the established in-repo pattern; adding two SDKs would be the first departure from it).

**Consequences:** `llmClient.ts`'s failure classification (`LlmError`, reason: `quota_exceeded | provider_rate_limit | provider_error | malformed_response | timeout | unknown`) reuses `openrouterClient.ts`'s exported `AiFailureReason` type rather than duplicating a near-identical union (CLAUDE.md "no duplicated types"), but only maps `429`→`provider_rate_limit` and `>=500`→`provider_error` with confidence — Gemini/Anthropic's other 4xx codes (auth vs. quota vs. malformed request) are not verified against their docs the way OpenRouter's `402` was, so they fall to `unknown` rather than a guessed classification. No new runtime dependency was added. `design/architecture.md` §10, `design/api-reference.md` §3.1b, and `design/tech-stack.md` §3 document the new client and its env vars (`LLM_PROVIDER`, `LLM_MODEL`, `GEMINI_API_KEY`, `ANTHROPIC_API_KEY`).

### AD-33: Resume suggestions are versioned rows scoped to a resume version; applying them always creates a NEW resume version; long resumes are chunked, never truncated (merge-workspace Phase 3)

**Decision:** A new `resume_suggestions` table (migration `20260714000002_resume_suggestions.sql`) stores one row per `suggestResumeImprovementsAction` call: `resume_id` (the exact resume version it was generated against), `target_role`, `suggestions` (jsonb array of `{id, category, title, detail}`), `model`, `created_at`, and a nullable `applied_as_resume_id` set when a chosen subset is later applied. `applyResumeSuggestionsAction` never mutates `resumes.parsed_text` in place — it reuses the exact same atomic `set_active_resume` path `uploadResume.ts` already uses (`ResumeRepository.create()`), so the AI-rewritten text becomes a brand new, separately versioned `resumes` row, and the old version's text is preserved unchanged. That required widening `NewResume.contentHash` from `string` to `string | null` (no SQL migration needed — the column and RPC parameter were already nullable, only the TS-side contract was tightened before this phase) — an applied version has no backing uploaded file, so it must never carry its parent's `content_hash` forward (the sha256 parse-once cache, AD-30, looks up the *most recent* row by hash; a forwarded hash would make a future re-upload of the original file incorrectly cache-hit the AI-rewritten text instead of the actual extraction). `applyResumeSuggestions` rejects (no version created) if the suggestion set's `resumeId` doesn't match the resume being applied to — suggestions describe specific text in a specific version and must not be silently misapplied to a newer, different version. Both `suggest` and `apply` chunk `resume.parsedText` via a new `chunkText()` (`src/shared/infrastructure/text.ts`, default 6000 chars/chunk) instead of truncating (jobhunt bug #2 — `enhance.py` hard-slices `working_text[:12000]`, silently discarding the rest of longer resumes) — every chunk is sent to the LLM and results are merged (suggestions concatenated + renumbered) or concatenated in order (rewritten text). Any provider failure on any chunk aborts the whole call before anything is persisted — there is no partial-suggestion-set or partial-rewrite fallback.

**Rationale:** This directly implements the merge plan's Phase 3 bullets ("Resume suggestions stored as versioned rows", "Apply creates a NEW version — never overwrite", "No truncation — chunk long resumes, jobhunt bug #2"). Reusing `ResumeRepository.create()`/`set_active_resume` rather than adding a second insert path follows the Phase 1 lesson (extend, don't duplicate) — the versioning semantics (atomic deactivate-previous-activate-new, monotonic `version`) are exactly what "creates a NEW version" already means in this codebase, so a parallel mechanism would be a needless duplicate. Chunk-then-concatenate (rather than chunk-then-merge-with-cross-chunk-context) was chosen because each chunk is self-contained resume text and the improvements being applied (rephrasing, adding metrics, restructuring) don't require cross-chunk coordination to apply correctly — a simpler, deterministic reassembly (`chunks.join("\n\n")`) that needed no new architecture. Aborting the whole call on any chunk failure (rather than falling back to that chunk's original text) was chosen over a partial-success design because the merge plan didn't ask for one, and a resume version is more valuable whole-or-absent than silently half-AI-rewritten with no signal to the user about which half.

This phase also surfaced a real conflict with `design/scope.md` §4's existing "Resume generation / editing | Platform assists skill tagging only" out-of-scope line, which Phase 3's explicit ask directly contradicts. Resolved the same way AD-31 reconciled AD-07/AD-26 against Phase 2's narrower ask: the merge plan is a standing, user-approved cross-repo plan, not a preemptive addition invented this session, so its explicit Phase 3 scope supersedes that line. `scope.md` §4 now lists "Auto-apply / auto-send applications" instead (carried over from jobhunt-app's own "no auto-apply" design rule, which this feature already respects — the user always reviews suggestions and explicitly chooses which to apply), and a new §3 P1.12 entry documents what's actually shipped.

**Alternatives considered:** Overwriting `resumes.parsed_text` in place on apply, matching jobhunt's own `apply_suggestions()` behavior (rejected — explicitly what the merge plan's "never overwrite" bullet forbids, and would break `job_scores.resume_version`'s existing "scores are keyed to an immutable resume version" invariant, AD-09/resume-versioning migration); a second `INSERT`-only write path for AI-applied versions instead of widening `NewResume.contentHash` (rejected — would duplicate `set_active_resume`'s atomic activate/deactivate logic for no benefit, see Rationale); carrying the parent resume's `content_hash` forward into the applied version (rejected — corrupts the parse-once cache's hit semantics, see Decision); truncating resume text like jobhunt does (rejected — explicitly the bug the merge plan calls out); a partial-success fallback that keeps a chunk's original text when its rewrite call fails (rejected — see Rationale; also would silently produce a resume that's part-AI-rewritten and part-original with no marker distinguishing the two, worse than a clean failure); building a full `/resume` UI for suggest/apply this phase (deferred — same "backend now, UI later" call as Phase 2's `embedding_score`/AD-31, since the merge plan's Phase 3 checklist is entirely backend-shaped and CLAUDE.md's "domain/application/infrastructure... before UI" ordering explicitly allows this).

**Consequences:** `design/limitations.md` §2.5 documents the no-UI-yet gap and the "stale suggestion set" failure mode. `design/erd.md`, `design/api-reference.md` (§1 two new server actions, §3.1b new external APIs, §4 `set_active_resume`'s now-nullable `p_content_hash`), `design/architecture.md` §10, `design/technical-design.md` §7, `design/scope.md` §3/§4, `design/security.md` §2/§5, `design/tech-stack.md` §3, `design/use-cases.md` (new UC-05a), and `.env.example` were all updated per CLAUDE.md's document-maintenance table. No backfill was needed (this is a wholly new table, no pre-existing rows to migrate).

### AD-34: Application drafting is a human-reviewed draft → edit → mailto flow, never an automated send; pending drafts are surfaced by reusing the existing `TelegramSender` port, not a new notification channel (merge-workspace Phase 4)

**Decision:** A new `src/features/applications/` module (domain/application/infrastructure/actions.ts, ported from jobhunt/apply.py's `draft()`/`mailto_link()`) implements the merge plan's Phase 4 checklist: a new `applications` table (migration `20260714000003_applications.sql`, `UNIQUE (job_id, kind)`, `status` one of `draft|sent|dismissed`), `draftApplicationAction`/`getApplicationForJobAction`/`updateApplicationContentAction`/`markApplicationSentAction`/`markApplicationDismissedAction` server actions, and a `/dashboard` review dialog (`ApplicationDraftDialog.tsx`) wired into both `JobRow` and `JobCard`. `LlmApplicationDraftProvider` calls the same provider-agnostic `llmClient`/`LLM_PROVIDER` abstraction AD-32 introduced (a third caller alongside resume suggest/apply), reusing jobhunt's `DRAFT_SYSTEM` prompt verbatim (truthful-only constraint, JSON-only response, 120-180/250-350 word targets by kind) and its context-length caps (job description 4000 chars, resume text 8000 chars — carried over as the same deliberate prompt-cost control AD-23 already established for stage-2 scoring, not a re-run of jobhunt bug #2, which was specifically about *merging* chunked output; a single application draft is one coherent call for one job, not a chunk-and-merge operation). `markApplicationSentAction` only ever records that the user opened a client-side `mailto:` link (`buildMailtoLink.ts`, a pure function ported from `mailto_link()`) themselves — no code path in this app calls an email-sending API. A `sent` row is terminal: `draftApplication.ts` refuses to redraft over it, and `updateApplicationContent`/`markApplicationSent`/`markApplicationDismissed` all reject any status other than `draft`. Redrafting a `dismissed` row is allowed (upsert on the `(job_id, kind)` conflict target resets `status` to `draft`). Pending drafts are surfaced by a new `notifyPendingDrafts()` (`src/features/applications/application/`), called once at the end of `scripts/notify.ts` after the existing job-match digest — it queries `applications` where `status = 'draft'`, formats a short HTML reminder in the same visual style as `formatDigestMessage.ts`, and sends it through the same `TelegramSender` port (`sendMessage`) that `sendDigest`/`sendNotification` already use.

**Rationale:** "mailto only for now" (the plan's literal wording) and jobhunt's own hard "no auto-apply / no bulk send" design rule (jobhunt's `CLAUDE.md`) both point the same direction: the app must never be the thing that actually sends an email. Modeling `sent` as a one-way, repository-enforced terminal state (not just a UI convention) makes that guarantee hold even if a future caller bypasses the dialog — the domain layer refuses the transition, not just the button being hidden. Reusing `TelegramSender` directly for the pending-drafts reminder (rather than folding it into `sendDigest.ts`'s internals, or building a parallel scheduling/notification-log mechanism) was chosen because the Phase 4 checklist's actual requirement is "surface... via existing digest infra" — the infra that matters is the Telegram delivery channel itself, not `sendDigest`'s specific job-matching/threshold logic, which has nothing to do with applications. This keeps the blast radius to one new call at the end of `notify.ts`'s `main()`, rather than threading an unrelated concept (draft applications) through the job-digest domain function's signature. No "already reminded" state was added — the reminder is a pure function of current `status = 'draft'` rows, so it is trivially correct (never stale, never needs invalidation) at the cost of repeating every cron run until the user acts, which is judged to be the right default for a "you have unfinished work" nag rather than a bug to suppress.

This phase also surfaced a real conflict with `design/scope.md` §4's existing "Automated job applications | Scope is discovery and triage, not application" and "Cover letter generation | Out of scope for V1" lines — Phase 4's explicit ask (draft + review + mailto-send, `kind` includes `coverletter`) contradicts both as literally written. Resolved the same way AD-31/AD-33 reconciled prior scope lines against their phases: the merge plan is a standing, user-approved cross-repo plan, not a preemptive addition invented this session, so its explicit Phase 4 scope supersedes those two lines. "Automated job applications" is narrowed in `scope.md` to describe what's still actually excluded — the app auto-*submitting* an application without a human sending it — since that is the real invariant jobhunt's design rule and the plan's "mailto only" phrasing both protect, not the mere existence of AI-assisted drafting. "Cover letter generation | Out of scope for V1" is removed as stale — `kind = 'coverletter'` is now a shipped, reviewable draft type, subject to the exact same human-review-before-send gate as `kind = 'email'`.

**Alternatives considered:** An SMTP send path guarded by an explicit confirm checkbox, matching jobhunt's own `send_email()` (rejected — the merge plan's Phase 4 line item explicitly says "mailto only for now," and SMTP credential storage/handling would be new attack surface this phase wasn't scoped to add); folding the pending-drafts reminder into `sendDigest`'s/`sendDigestMvp`'s message body or `DigestKeyboardOptions` (rejected — applications are not job matches, and coupling an unrelated concept into that function's parameters/return shape for one extra line of Telegram text was a worse fit than one new call in the composition root that already owns "things this cron run notifies about"); a soft-delete/undo window on `markApplicationSent` (rejected — not asked for, and "sent" being genuinely irreversible is the honest state of the world once a `mailto:` link has been opened); building an "already reminded" `applications`-adjacent tracking table to avoid repeat pending-draft nags (rejected — see Rationale; adds a new failure mode — a reminder that silently stops firing because of a stuck flag — for a problem the stateless design doesn't have).

### AD-35: JSearch and Adzuna are cron-driven `JobSourceScraper` adapters; the static careers-URL fetcher is a valid `JobSource` but a manual-trigger-only script, deliberately excluded from `JOB_SOURCES` (merge-workspace Phase 5)

**Decision:** `job_source` gains three new enum values (migration `20260715000001_jsearch_adzuna_careers_url_sources.sql`): `jsearch`, `adzuna`, `careers_url`. `JSearchScraper.ts` and `AdzunaScraper.ts` are registered in `registry.ts`/`sourceScrapers` like every other feed-based adapter — both are query/country search APIs (unlike the ATS boards), so each issues one request per `(search term, target country)` combo, capped at `MAX_SEARCH_TERMS = 2` to keep worst-case requests-per-run small against typically tight free-tier rate limits, following `MyCareersFutureScraper.ts`'s existing "issue N requests instead of one big feed" shape. Both auto-disable (clean skip, no error) when their API credentials (`RAPIDAPI_KEY`; `ADZUNA_APP_ID`/`ADZUNA_APP_KEY`) are unset, the same `validateWellfoundConfig`-style discriminated-union convention `WellfoundScraper.ts` established. Both fix jobhunt bug #4 — jobhunt/sources.py's `_normalize_jsearch` falls back to `job_apply_link` as the id when JSearch omits `job_id`, but apply/redirect links can carry per-request tracking tokens that change across identical re-fetches of the same posting; since `RawJob.sourceJobId` is the `(source, source_job_id)` dedup/upsert key the whole ingest pipeline relies on for "update in place, don't re-insert as new," both adapters reject (silently filter out, same convention as `RemoteOkScraper.ts`'s `isJobEntry`) any entry lacking a genuine id instead of substituting an unstable fallback.

`careers_url` is added to the `JobSource` TS union (and the Postgres enum) because `jobs.source` needs a real value for jobs ingested from it, but it is **not** added to `JOB_SOURCES` (`src/shared/domain/enums.ts`) — the curated subset `getSourceHealthReport` iterates for source-health tracking and `NotificationPreferences.sources` validates against. `computeSourceHealthSummary`'s `isStale` flag fires whenever `hoursSinceLastRun >= SOURCE_HEALTH_CONFIG.staleAfterHours` (default 6h) for any source that has ever run at all; a manual-trigger source with no expected run cadence would flip to a permanent, misleading "Stale — check the scrape workflow is still including this source" recommendation the very first time it was actually used. Excluding it from `JOB_SOURCES` keeps the health/notification-filter set scoped to sources with a genuine expected cadence, the same reasoning `SOURCES_REQUIRING_BOARD_TOKEN` already applies for a different curated subset of the same enum. `scripts/source-analytics.ts`'s separately-hardcoded `ALL_SOURCES` (a different, purely descriptive 30-day report with no staleness concept) does include all three new sources, since a manual source appearing with a low run count there is informational, not misleading.

The static careers-URL fetcher itself (`CareersUrlScraper.ts`'s `fetchCareersUrlJobs`, ported from jobhunt/sources.py's `fetch_company_careers`) is **not** a `JobSourceScraper` implementation — that interface's `fetchJobs(companies, roles)` shape assumes a registry-wide feed/board with no per-call target, and this is inherently single-URL-per-invocation. It gets its own small function plus a new `CareersPageExtractor` port (implemented by `LlmCareersPageExtractor`, a fourth caller of the Phase 3 `llmClient`/AD-32 abstraction, alongside resume suggest/apply and application drafting) and its own manual entry point, `scripts/scrape-careers-url.ts` (`npm run scrape:careers-url -- <url>`), which otherwise reuses the exact `tagLocations` → `hasAllowedLocation` → `ingestJobs` → `recordRun` pipeline `scrape.ts`'s per-source block already uses, so a careers_url run gets the same location filtering, cross-source dedup, and `scrape_runs` provenance as every cron source, just invoked for one URL instead of a registry loop. A synthetic, deterministic `sourceJobId` (`sha256(url|title)`, mirroring jobhunt's own `hashlib.sha256((link + title)...)[:24]`) stands in for the stable ID a real ATS API would provide, since a static page has no natural per-listing ID. The page's full HTML is chunked (`chunkText`, not truncated) before each chunk is sent to the LLM extractor and results are concatenated — the same "don't silently drop the tail of a long document" principle AD-33 established for resumes (jobhunt bug #2), applied here even though the plan didn't explicitly call out a truncation bug for this source, because the util already existed and reusing it was strictly better than truncating at a fixed char cap like jobhunt's own `text[:15000]`.

**Rationale:** Adzuna's own covered-country list does not include the UAE — only India and Singapore of this platform's three target regions are reachable through it (default `ADZUNA_COUNTRIES=in,sg`, documented in `design/limitations.md` §1.1a rather than silently omitted or worked around with an unsupported country code that would just 404). `stripHtml` (`src/shared/infrastructure/text.ts`) was extended to also strip `<script>/<style>/<noscript>` block *content* (not just their tags) before this phase, not just for the careers-URL fetcher's benefit — every existing ATS-description caller is unaffected (job descriptions never contain those tags) but the careers-URL fetcher is the first caller to feed it a *full page* of arbitrary HTML, where leaking script/style body text into the LLM prompt would waste tokens and confuse extraction the way jobhunt/sources.py's own `_html_to_text` explicitly guards against. Reusing this one util instead of writing a second HTML-to-text function keeps CLAUDE.md's "no duplicated logic" rule intact.

**Alternatives considered:** Forcing `CareersUrlScraper` into the `JobSourceScraper` interface with the URL read from an env var (rejected — would make a fundamentally single-target, manual operation look like a registry-wide cron source, and the "one URL per env var" shape doesn't support fetching a different company's page without an env var edit + redeploy, worse than a CLI arg); including `careers_url` in `JOB_SOURCES` for completeness (rejected — see Decision; the staleness false-positive is a real, immediate usability problem, not a hypothetical); truncating the careers page text at a fixed char cap like jobhunt's `[:15000]` (rejected — `chunkText` already existed for exactly this "don't drop content silently" reason and reusing it cost nothing extra); a real ATS-style probe/validation check for JSearch/Adzuna in `validate-sources.ts` (rejected — that script probes *board tokens* the user configures per-company; JSearch/Adzuna have no per-company config to probe, only API credentials, which `sourceFallbackConfigCheck` — extended this phase to cover both — already checks for a "credentials set but explicitly disabled" contradiction, the same class of check it already did for Wellfound).

### AD-36: One-off jobhunt-app cutover script uses Node's built-in `node:sqlite`, not a new dependency; only jobhunt jobs the user actually reviewed are migrated, marked already-notified rather than re-alerted (merge-workspace Phase 6)

**Decision:** `scripts/migrate-jobhunt-sqlite.ts` (`npm run migrate:jobhunt-sqlite -- <path-to-jobhunt.db>`) reads jobhunt-app's SQLite database with Node 24's built-in `node:sqlite` (`DatabaseSync`, `readOnly: true`) rather than adding `better-sqlite3`/`sqlite3` as a new dependency — verified against `node_modules/@types/node/sqlite.d.ts` (types ship in the already-installed `@types/node@^25.9.3`) and a real read against a scratch fixture db before relying on it, per the Phase 0 lesson about re-verifying gating findings. **Resumes:** every jobhunt `resumes` row is migrated oldest-first via `ResumeRepository.create()` (the same `set_active_resume` path `uploadResume()` uses), using jobhunt's own already-extracted `working_text` (falling back to `raw_text`) directly rather than re-parsing — jobhunt already did that work, and re-running pdf-parse/mammoth over it would be pure waste. Skills are re-derived via job-scraper's own `extractSkills`/`SKILLS_DICTIONARY` rather than carrying over jobhunt's differently-shaped `profile_json` skills, so the migrated rows score consistently with everything scraped natively. Because every `set_active_resume` call is an unconditional new version (AD-30), this makes the most-recently-created jobhunt resume the active one post-migration — the closest available proxy for "the one the user was actually using," since jobhunt never persisted a separate "currently selected" flag (`app.py`'s resume picker is a Streamlit session-state concept only, not a DB column). The original file bytes are copied into Storage (dedup path `<sha256>.<pdf|docx>`, same as a live upload) on a strictly best-effort basis: jobhunt stores `file_path` relative to its own CWD (`data/...`), not relative to the sqlite file, so the script tries the project root implied by the db's own location first, then the db's directory; a missing file, a `.txt`/`.md` resume (jobhunt supports these, this app doesn't), or a failed upload all fall back to migrating the text only, with `content_hash: null` and a `legacy-jobhunt/resume-<id>.<ext>` placeholder path — never a hard failure, since the text is the only part that actually feeds scoring. **Jobs:** only rows with `jobs.status != 'new'` (jobhunt's own vocabulary: `drafted`/`applied`/`skipped`) are migrated — a `new` row was never reviewed in jobhunt either, so there is nothing to protect against re-surfacing. Recognized rows (`source` maps `jsearch`→`jsearch`, `careers`→`careers_url`, both already-valid `JobSource` values from AD-35; and a non-empty `job_id`/`title`/`url`) go through the exact same `tagLocations` → `hasAllowedLocation` → `ingestJobs` pipeline `scrape.ts`/`scrape-careers-url.ts` use, then every resulting job id is passed to `NotificationRepository.markManyNotified` — `notifications_log` has no role/resume scoping, so this permanently suppresses a fresh notification for a job the user already handled, regardless of which role selection or resume version scores it later.

**Rationale:** The plan's literal phrase was "resumes + seen jobs → Supabase" — "seen" reads naturally as jobhunt's own status vocabulary (`!= 'new'`), not "every job jobhunt ever fetched," which would just re-import noise with no notification-suppression benefit (a `new` job was never shown to the user as reviewed, so there's no duplicate-alert risk to prevent). Routing seen jobs through the standard ingest pipeline (not a raw insert) was chosen over a bespoke insert path because it already handles everything a raw insert would have to reimplement — cross-source fingerprint dedup, location tagging/filtering, salary/contact/attribute extraction — and reusing it costs nothing extra (Phase 5's `scrape-careers-url.ts` established the same "reuse the pipeline for a one-off entry point" precedent). Not writing a `scrape_runs` row for this script (unlike every real scrape) is deliberate: this isn't a source poll, and a synthetic "scrape run" for a one-time historical import would pollute `getSourceHealthReport`'s cadence/staleness math for `jsearch`/`careers_url` with a data point that doesn't reflect either source's actual health.

**Alternatives considered:** `better-sqlite3` (rejected — `node:sqlite` already does everything this one-off script needs with zero new dependency/`npm audit` surface, and Node 24 — this project's pinned runtime — ships it without an experimental flag); migrating *all* jobhunt jobs regardless of status (rejected — see Rationale; also would inflate `jobs`/`job_duplicates` with rows carrying no salary/description quality guarantees jobhunt itself didn't apply); a raw `INSERT`/upsert straight into the `jobs` table bypassing `ingestJobs` (rejected — see Rationale; would silently skip location filtering and fingerprint dedup, letting a jobhunt-era job the location filter would otherwise reject sit in the table forever with no `locationTags`, which `validateNormalizedJob` explicitly forbids); treating the migration as idempotent for resumes by skipping rows whose content already exists (rejected — no reliable dedup key exists for jobhunt resumes with no backing file on disk, and `set_active_resume`'s existing contract is already "every call is a new version, cache hit or miss" per AD-30 — fighting that contract for one script would be inconsistent with the live upload path's own behavior); writing a `scrape_runs` row tagged `careers_url`/`jsearch` for this run (rejected — see Rationale).

**Consequences:** `MAX_SEARCH_TERMS = 2` for both aggregator adapters means broad role selections (more than 2 expanded roles) are not exhaustively searched per run for these two sources specifically — a deliberate cost/completeness tradeoff, not a bug, and consistent with the "small, predictable request volume over exhaustive coverage" call this AD makes explicitly. The careers-URL fetcher's extraction quality is only as good as the configured LLM and the page's HTML structure — a heavily JS-rendered careers page will yield zero results, the same static-HTML-only limitation jobhunt-app's own reference implementation states and does not attempt to work around (no headless browser was added).

**Consequences:** `design/erd.md` (new `APPLICATIONS` entity + two relationships + two `Key Constraints` rows), `design/api-reference.md` (new "Applications" §1 subsection, five server actions), `design/architecture.md` §10 (llmClient's caller list), `design/security.md` §2 (new RLS row), `design/scope.md` §3 (new P1.13 entry) and §4 (two lines reconciled per above), `design/use-cases.md` (new UC-17/UC-18, two new user stories), and `design/user-guide.md` were all updated per CLAUDE.md's document-maintenance table. `src/features/jobs/domain/JobRepository.ts`/`SupabaseJobRepository.ts` gained a `getById(id)` method — `findForDashboard`'s page rows omit `description` (P1 #4) to shrink the dashboard payload, but application drafting needs the full posting text server-side, and no existing method returned a single full `Job`. No backfill was needed (wholly new table).

### AD-37: Resume version restore reactivates content via the existing `set_active_resume` path, never flips `is_active` on an old row in place (post-audit closure)

**Decision:** `ResumeRepository` gains `listVersions()` (every resume row, newest version first — no new SQL, `select * from resumes order by version desc`). A new application function, `restoreResumeVersion(resumeId, deps)`, looks up the target row via `listVersions()`, rejects if it's missing or already active (`DomainValidationError`), and otherwise calls the existing `resumeRepository.create()` — the same `set_active_resume` RPC every upload and every suggestion-apply already goes through — seeded with the target version's exact `filePath`/`parsedText`/`skills`/`contentHash`. This produces a brand new, active version with byte-identical content to the one being "restored"; the old row is untouched. `restoreResumeVersionAction` (`src/features/resume/actions.ts`) and a `ResumeVersionHistory` client component (`src/components/resume/`, following `SkillsEditor.tsx`'s exact `useTransition`/`ActionResult` pattern) wire this into `/resume` behind a version-history card that only renders when more than one version exists.

**Rationale:** Audit finding #1 was that old resume versions were preserved in Postgres (`set_active_resume` deactivates, never deletes) but had no reachable undo path — `ResumeRepository` had no `listVersions`/`reactivate`, and `resume/page.tsx` only ever called `getActive()`. A literal "reactivate" (`update resumes set is_active = true where id = ...`) was rejected because every other write path in this module treats a resume version as immutable once created — `job_scores.resume_version` and `resume_suggestions.resume_id` both key history off a specific version id/number (AD-30, AD-33), and AD-33 went out of its way to make "apply" create a new version specifically so the old one's text is "preserved unchanged." Toggling `is_active` in place on an old row wouldn't corrupt that data, but it would break the pattern's consistency for no benefit: `set_active_resume` already is the one atomic "make this content the active version" operation, and reusing it for restore means restore gets the exact same monotonic-version-increment and deactivate-previous guarantees a fresh upload gets, with zero new SQL.

**Alternatives considered:** A new `reactivate_resume_version(id)` RPC that flips `is_active` on the existing row instead of inserting a new one (rejected — see Rationale; also would need `resumes.version` to stay meaningful as "the order versions were created," which an in-place reactivation would violate the first time a restored version was later superseded by a new upload, since the old row's `version` number would then be reused as if it were newly created); a `getById(id)` repository method instead of filtering `listVersions()`'s result client-side (rejected — the expected version count for a single-user resume history is small, and `listVersions()` already fetches every row, so a second query for one more row lookup would be pure overhead for no correctness benefit at this scale); showing full version history unconditionally instead of gating the card on `versionSummaries.length > 1` (rejected — a brand-new user with exactly one resume has nothing to restore, so the card would only ever show a single disabled "Active" row, adding UI noise with no action available).

**Consequences:** `ResumeRepository`'s two existing test-fake construction sites (`uploadResume.test.ts`, `applyResumeSuggestions.test.ts`) needed `listVersions: vi.fn()` added to stay structurally typed against the widened interface — a mechanical, one-line change at each site, not a behavior change. `design/api-reference.md`, `design/scope.md` (new P1.15 entry), and `design/user-guide.md` were updated per CLAUDE.md's document-maintenance table. No migration needed — `listVersions` and the restore flow both operate entirely on `resumes` columns/RPC that already existed.

**Decision:** Two audit findings (#2, #3) identified fully-built, fully-tested backend capability with zero UI callers. Per the user's explicit choice (both: "wire minimal UI now"), both were wired with the smallest UI that makes the existing actions reachable, no new backend logic:

### AD-39: `set_active_resume` restored to `returns setof resumes` — two later migrations had silently regressed it back to a bare composite return (bugfix session, MERGE_PLAN.md Bug 2)

**Decision:** `20260715000002_fix_set_active_resume_setof_regression.sql` drops and recreates `set_active_resume` (still the 4-arg, content-hash-aware signature) as `returns setof resumes`, matching `database.types.ts`'s generated `Returns: ...[]`/`SetofOptions.isSetofReturn: true` and `SupabaseResumeRepository.create`'s `data?.[0]` unwrap.

**Rationale:** Real-world testing hit `"set_active_resume returned no row"` on restore. Tracing the full path (UI action → `restoreResumeVersion` → `SupabaseResumeRepository.create` → RPC → migrations) found that `20260612000006_fix_rpc_return_types.sql` had already fixed this exact class of bug once — changing the function to `returns setof resumes` so PostgREST serializes an array — but two later migrations that added unrelated columns (`20260618000002_resume_versioning.sql`'s version-number column, `20260710000001_resumes_content_hash.sql`'s content-hash column) each had to drop+recreate the function to change its signature/body, and both recreated it as a bare `returns resumes` instead, neither migration's own comments mentioning the return type at all — silently re-introducing the regression. Verified directly against a local Postgres 16 (not mocked): `select proretset from pg_proc where proname = 'set_active_resume'` was `f` on the pre-fix migration chain and `t` after this migration; a `row_to_json` of the pre-fix function's result is a bare JSON object (`{"id": ..., ...}`), while `json_agg` of the post-fix function's result is a JSON array (`[{"id": ..., ...}]`) — exactly the shape PostgREST hands to `data?.[0]`. Also confirmed the underlying `INSERT` succeeds either way (the row was visible in `resumes` immediately after the "no row" error was thrown pre-fix) — this was purely a return-shape bug, not a transaction/RLS issue. RLS was checked and ruled out: the function is `SECURITY INVOKER` (no `security definer` clause), and `resumes`' `authenticated_full_access` policy (`using (true) with check (true)`) permits the `UPDATE`/`INSERT`/`RETURNING` regardless.

**Alternatives considered:** Changing `SupabaseResumeRepository.create` to accept either an object or an array (rejected — would paper over the SQL/types mismatch instead of fixing it, and the generated `database.types.ts` already correctly declares the setof/array contract; the SQL should conform to the types, not the other way around); a CI check diffing every migration's function `returns` clause against `database.types.ts` (considered as a longer-term follow-up, not implemented this session — out of scope for a same-day bugfix, and no such generator/lint step exists yet in this codebase to hang it off of).

**Consequences:** `design/api-reference.md` §4 and `docs/repositories.md` §3 updated to state `RETURNS SETOF resumes` explicitly (the former had drifted to document the broken bare-`resumes` shape as if correct — now corrected and annotated with this AD so it can't happen a third time without a code comment pointing at it). `SupabaseResumeRepository.test.ts` gained a regression test pinning the client-side symptom (a bare-object RPC response still throws the same clear error rather than silently misbehaving). No integration test against a live PostgREST instance was added to the automated suite — this sandbox has no `pg` client dependency, no Docker/Supabase-CLI-local available, and adding a new runtime dependency plus CI database provisioning for one regression test was judged disproportionate; the manual local-Postgres verification above (`proretset`, `row_to_json` vs `json_agg`) is documented here and in the migration file's own comment instead, and should be re-run the same way if this function's `returns` clause is ever touched again.

### AD-40: `uploadResume()` now parses+validates before touching Storage or the DB, and cleans up Storage on a DB-insert failure (bugfix session, MERGE_PLAN.md Bug 1a)

**Decision:** `actions.ts`'s `uploadResumeAction` no longer uploads to Supabase Storage itself before calling `uploadResume()`. A new `ResumeStorage` port (`src/features/resume/domain/ResumeStorage.ts`, implemented by `SupabaseResumeStorage`) is injected into `uploadResume()` alongside `ResumeRepository`. The application function's order is now: (1) check the sha256 parse-once cache, (2) parse + `validateParsedText` if not cached, (3) extract/validate skills, (4) `resumeStorage.upload(...)`, (5) `resumeRepository.create(...)` (the `set_active_resume` RPC) — wrapped so that if step 5 throws, `resumeStorage.remove(...)` is called (best-effort, swallowing its own errors) before re-throwing the original error.

**Rationale:** Real-world testing reported "Invalid PDF structure" on upload, with the resume nonetheless appearing in version history after a reload. Tracing `uploadResumeAction` end-to-end found the previous order was Storage upload → parse/validate → DB insert — a parse failure after a successful Storage upload left an orphaned Storage object with no `resumes` row pointing at it (never surfaced to the user as a visible bug, since `listVersions()` reads the DB table, not Storage, but real waste and a latent inconsistency). Reordering so nothing is persisted anywhere until parsing and validation both succeed removes that failure mode entirely, and matches this session's other finding (AD-39) that a DB insert can independently fail *after* actually succeeding (the RPC return-shape bug) — the Storage-cleanup-on-DB-failure step specifically guards against stranding a Storage object in that scenario too, not just the parse-failure case.

**Alternatives considered:** Wrapping the whole thing in a Postgres transaction spanning Storage + DB (rejected — Supabase Storage operations are not transactional with Postgres DML, the same reason AD-09's `set_active_resume` RPC exists for the DB-only atomicity it *can* provide); moving the Storage upload call inline into `uploadResume.ts` without a new port/interface (rejected — would import a Supabase Storage client directly into the application layer, violating the repository-pattern/dependency-direction rules this codebase already enforces for `ResumeRepository`; a port is the same amount of code and keeps `uploadResume.test.ts` mockable without a real Storage client, matching every other application function's existing testing convention); silently swallowing a DB-insert failure after Storage succeeds and returning a "partial success" result (rejected — the caller needs to know the upload failed; a stranded-but-recoverable Storage object is a much smaller problem than telling the user "success" when the resume isn't actually active).

**Consequences:** `UploadResumeDeps` gained a required `resumeStorage: ResumeStorage` field — the one existing construction site (`actions.ts`) was updated in the same commit; `uploadResume.test.ts`'s `makeDeps()` helper gained a `makeResumeStorage()` mock, used by five new regression tests (parse-failure ⇒ no Storage/DB writes; validation-failure ⇒ no Storage write; Storage-before-DB-insert ordering; DB-insert-failure ⇒ Storage cleanup; cleanup-failure doesn't mask the original error). `scripts/sweep-stranded-resumes.ts` (`npm run sweep:stranded-resumes`) was added as a one-time, read-only-by-default sweep for Storage objects orphaned by the pre-fix ordering (and a defensive check for any `resumes` row with suspiciously short `parsed_text`, though none should be reachable post-fix or even pre-fix, since `validateParsedText` already ran before every `resumeRepository.create()` call in this codebase's history — no evidence of any such row was found when reasoning through the code paths, but the check costs nothing to leave in place for a human to run and confirm). No SQL/schema change.

### AD-41: PDF text extraction swapped from `pdf-parse` to `pdfjs-dist` (bugfix session, MERGE_PLAN.md Bug 1b)

**Decision:** `src/features/resume/infrastructure/parsePdf.ts` now uses `pdfjs-dist`'s Node ("legacy") build (`pdfjs-dist/legacy/build/pdf.mjs`, `getDocument()` + per-page `getTextContent()`) instead of the `pdf-parse` package, which is removed as a dependency (along with its ambient type declaration, `src/types/pdf-parse.d.ts`). `next.config.ts`'s `serverExternalPackages` updated from `["pdf-parse"]` to `["pdfjs-dist"]`. The public `parsePdf(buffer): Promise<string>` contract (and its place behind the existing `ParseResumeText`/`parseResumeFile` port) is unchanged, so no caller (`uploadResume.ts`, `actions.ts`) needed any change.

**Rationale:** Real-world testing hit `"Invalid PDF structure"` on a real PDF upload. `pdf-parse@1.1.1` (last published years ago, effectively unmaintained) bundles its own old, pinned internal fork of PDF.js rather than depending on the actively-maintained `pdfjs-dist` package directly — that pinned fork is a well-documented source of exactly this error message for real-world PDFs using newer PDF-spec features (incremental updates, PDF 1.7+/2.0 xref streams, certain encrypted-but-empty-password documents) that its outdated parser doesn't recognize. The specific failing PDF was not available as a fixture at the time this fix was made, so this decision proceeds on `pdf-parse`'s known, documented brittleness rather than a diagnosis of that exact file — a maintained, actively-updated parser is very likely to resolve any file that trips this class of bug regardless of its exact contents. If the original failing PDF is later supplied and still fails against `pdfjs-dist`, that would point to a genuinely different root cause needing its own investigation. `pdfjs-dist` is Mozilla's own PDF.js, published continuously (this session installed v4.10.38, current at time of writing), and ships an explicit Node-targeted "legacy" build for exactly this non-browser use case.

**Alternatives considered:** Patching/forking `pdf-parse` to update its internal PDF.js fork (rejected — `pdf-parse` is a thin wrapper with no meaningful logic of its own beyond that bundled fork; forking it would mean maintaining a fork of a fork for no benefit over depending on `pdfjs-dist` directly); adding a second PDF library as a fallback when the primary throws (rejected — unrequested complexity, and untestable without real-world failing fixtures for both libraries; a single well-maintained parser is simpler and directly addresses the root cause); keeping `pdf-parse` and only catching-and-retrying on `"Invalid PDF structure"` (rejected — retrying the same broken parser against the same bytes produces the same failure, this isn't a transient error).

**Consequences:** `pdf-parse` removed from `package.json` dependencies; `pdfjs-dist@^4.10` added. `design/tech-stack.md` §1/§5 updated. `parsePdf.test.ts` added (previously untested — `pdf-parse` had no unit test in this codebase, likely because constructing a valid fixture by hand was judged not worth it at the time) with a hand-written, byte-exact-offset minimal PDF fixture (mirroring `parseDocx.test.ts`'s "build a real fixture, no binary checked in" convention) covering both a valid single-page extraction and a non-PDF-buffer failure case; both were verified to pass against the new parser locally before this commit. No behavior change to `validateParsedText`, the sha256 parse-once cache, or any other part of the upload pipeline — this is a drop-in infrastructure-layer swap.

### AD-42: `llmClient.ts`'s default provider routes through the existing OpenRouter client/key instead of requiring a second provider key (bugfix session, MERGE_PLAN.md Issue 3)

**Decision:** `LlmProvider` gains a third value, `"openrouter"`, which is now the default (`currentLlmProvider()`'s fallback, replacing `"gemini"`). `openrouterClient.ts` gained a new exported `callOpenRouterCompletion()` — a plain (non-schema-constrained) chat completion, sharing a new internal `sendOpenRouterRequest()` helper with the existing `callOpenRouterJson()` so the endpoint/auth/retry/failure-classification logic isn't duplicated (CLAUDE.md "no duplicated logic"). `llmClient.ts`'s new `callOpenRouter()` branch calls `callOpenRouterCompletion()` with `request.system`/`request.user` mapped to `{role: "system"}`/`{role: "user"}` chat messages, `jsonMode` mapped to OpenRouter's loose `response_format: {type: "json_object"}` (not a strict schema — callers already parse leniently via `lenientJson.ts` regardless), and the model defaulting to `google/gemini-2.5-flash` (`DEFAULT_MODELS.openrouter`, overridable via the existing `LLM_MODEL` env var). `OpenRouterError`s thrown from that path are now mapped to `LlmError` preserving the original `reason` (previously any non-`LlmError` exception fell through to a generic `"unknown"` reason). `"gemini"`/`"anthropic"` (direct REST to those providers, unchanged code) remain selectable via `LLM_PROVIDER` for anyone who wants `llmClient.ts`'s callers (resume suggestions, application drafts, careers-page extraction) on a different key/provider than scoring.

**Rationale:** Real-world testing hit drafting failing with `"Missing required environment variable: GEMINI_API_KEY"` — a second AI provider key was required just to use application drafting, on top of the `OPENROUTER_API_KEY` already required for job scoring. The user's explicit decision (per this session's task) was to route drafting through the *existing* `openrouterClient.ts` using model `google/gemini-2.5-flash`, rather than adding retry/fallback logic or making `GEMINI_API_KEY` conditionally-required in some other way — one env key now powers both scoring and drafting/suggestions by default. This *narrows* AD-32's original "Gemini Flash default, Anthropic optional" decision for the default case only; AD-32's core reasoning for keeping `llmClient.ts` and `openrouterClient.ts` as two separate *abstractions* (different call shapes: strict-schema job scoring vs. free-text/lenient-JSON resume coaching and application drafting) still holds and is unchanged — only the default *transport* for the free-text abstraction changed, from a second REST provider to the same OpenRouter gateway already in use, via a second exported function in the same client module rather than a new one.

**Alternatives considered:** Making `GEMINI_API_KEY` optional and falling back to a hardcoded default/no-op when unset (rejected — would silently degrade drafting to a broken state instead of working via the key the user already has configured); merging `llmClient.ts` into `openrouterClient.ts` entirely, deleting the Gemini/Anthropic direct-REST code paths (rejected — not asked for, and removes a working, tested escape hatch for anyone who does want a distinct key/provider for this feature; AD-32's schema-vs-free-text rationale for keeping two abstractions is unaffected by which transport the free-text one defaults to); duplicating a second copy of the OpenRouter request/retry/error-classification logic inside `llmClient.ts` instead of exporting `callOpenRouterCompletion` from `openrouterClient.ts` (rejected — directly violates CLAUDE.md's "no duplicated logic," and `openrouterClient.ts` is already the one place `OPENROUTER_API_KEY`/the OpenRouter endpoint is used, per `architecture.md` §5's infrastructure-layer-only-touches-external-clients rule).

**Consequences:** `.env.example` updated — `GEMINI_API_KEY` moved out of the "Required" section (now commented/optional, only needed for `LLM_PROVIDER=gemini`); `OPENROUTER_API_KEY`'s comment now notes it powers both scoring and drafting/suggestions by default. `design/tech-stack.md` §3, `design/security.md` §5, `design/architecture.md` §10, `design/api-reference.md` §3.1b, `design/scope.md` §3, `design/use-cases.md` (UC precondition + resume-coaching use case) all updated to describe `openrouter` as the default and `gemini`/`anthropic` as opt-in. `llmClient.test.ts` restructured: the former "gemini (default provider)" describe block is now "gemini (LLM_PROVIDER=gemini)" (explicitly sets the env var, since it's no longer the default), and a new "openrouter (default provider)" describe block covers the new default path including `LLM_MODEL` override, `jsonMode` → `json_object`, and `OpenRouterError` reason preservation (402 → `quota_exceeded`). `openrouterClient.test.ts` gained a `callOpenRouterCompletion` describe block. No change to `AiScoreProvider`/`OpenRouterAiScoreProvider` (job scoring) — `callOpenRouterJson`'s behavior and callers are untouched.
